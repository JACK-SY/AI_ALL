## Q-01：平台资源池的"租户可选模型池"形成机制

**质询内容**：租户可选模型池的形成机制，是推式（发布时指定租户范围）、模板式（分级套餐驱动）、还是首版仅从公共接入点候选反推的懒计算？

**确认结果**：✅ **推式**——平台管理员在模型发布时显式指定租户范围。

**测试影响**：测试需覆盖发布向导中租户范围选择器的功能验证，包括多租户选择、全选/反选、重复发布时的租户范围叠加等场景。

## Q-02：模型发布与资源池投放的操作合并关系

**质询内容**：模型发布、加入资源池、生成公共接入点是同一个向导流程一次完成，还是发布后再独立操作？

**确认结果**：✅ **同一向导流程一次完成**，后续可独立调整。

**测试影响**：测试需验证向导完整流程（发布→选租户→可选生成接入点→可选安全护栏）的端到端场景，以及向导各步骤的前进/后退/取消行为。同时需覆盖向导完成后独立调整资源池范围和接入点配置的场景。

## Q-03：默认启用策略的配置粒度

**质询内容**：默认启用策略的配置粒度是资源级（每个租户×每个资源独立设置）、租户级（同一租户所有新资源统一策略）、还是平台全局级？

**确认结果**：✅ **资源级**——每个租户×每个资源独立设置。

**测试影响**：测试需构造"同一模型分配给不同租户设置不同默认启用策略"的交叉场景，验证策略独立性。需覆盖：默认启用场景下租户资源自动开启、严格模式场景下租户资源默认关闭、向导中策略覆盖功能。

## Q-04：资源关闭失效绑定的"自动恢复"语义

**质询内容**：重新开启资源后，历史失效绑定是立即无条件自动恢复、恢复但需确认、还是可选恢复？

**确认结果**：✅ **立即无条件自动恢复**，同步触发 API Key 权限更新、转发侧 token 同步和缓存清理。

**测试影响**：测试需验证完整链路：关闭资源→绑定失效→调用被拒→重新开启→绑定自动恢复→调用恢复成功。需验证权限同步、转发侧 token 更新、Redis 缓存清理的联动正确性。需关注恢复操作的原子性和并发场景。

## Q-05：公共接入点从租户资源池"移出"的语义

**质询内容**：公共接入点从租户资源池"移出"，是删除资源池记录（清除绑定）、等效于强制关闭（保留失效绑定）、还是本期不支持移出？

**确认结果**：✅ **等效于强制关闭**——保留失效绑定，走与关闭资源相同的链路。

**测试影响**：测试需验证"移出"操作与"租户管理员手动关闭"在效果上一致：资源不可见、API Key 绑定进入失效状态、调用被拒绝。需额外验证操作来源标识（平台管理员移出 vs 租户管理员关闭）在审计日志中的区分。

**确认结果**：✅ **模型供应商级**（`providerId + modelId`），同一供应商下多渠道由转发侧自行处理。

## Q-12：API Key 候选选择器中已关闭/失效模型候选项的展示

## Q-13：租户管理员关闭模型后，通过公共接入点是否仍可命中该模型（待确认）

**质询内容**：租户管理员关闭模型 M 后，通过已启用公共接入点（候选包含模型 M）调用时，模型 M 是否仍可被命中？

**当前分析**：文档产品规则第6条明确"关闭模型不影响已开启公共接入点的内部路由"，暗示仍可命中。但从测试角度，这是一个关键的交叉场景——模型级关闭但接入点级仍可用，用户可能困惑于"我明明关闭了这个模型为什么还能调用到"。

**确认结果**：✅ **可以命中**——公共接入点视为独立资源，关闭模型不联动影响已启用公共接入点的内部路由。

**测试影响**：测试必须构造"模型级关闭+接入点级仍可用"的交叉场景，验证：（1）通过公共接入点调用仍可命中该模型（2）调用日志中记录的实际命中模型为模型 M（3）模型广场不可见模型 M、模型直连调用被拒绝（4）用户认知一致性——同一模型在不同调用路径下行为不同。此为高风险测试场景。

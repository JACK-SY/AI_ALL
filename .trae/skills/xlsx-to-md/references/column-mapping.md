# 列映射与配置参考

本文件定义了 Excel 测试用例到 Markdown 标准字段的双向映射规则，以及如何通过 config.json 灵活适配不同项目。

## 标准字段与别名映射

config.json 的 `column_aliases` 字段控制列名匹配逻辑：

| 标准字段 | 默认别名（按优先级） | 必填 | 说明 |
|---------|---------------------|------|------|
| 用例编号 | 用例编号 / 测试用例编号 / TC编号 / Case ID / case_id / 编号 | ✅ | 唯一标识，缺失将中止转换 |
| 模块 | 模块 / 功能模块 / Module / module / 所属模块 | ✅ | 用于分组，子模块名自动合并 |
| 用例标题 | 用例标题 / 标题 / 测试点 / TestCase / title / 测试用例名称 | ✅ | 用例核心描述 |
| 优先级 | 优先级 / Priority / priority / 级别 | ✅ | 枚举值由 `priority_levels` 定义 |
| 前置条件 | 前置条件 / 前提条件 / Precondition / precondition / 前置 | ❌ | 引用块展示 |
| 测试步骤 | 测试步骤 / 步骤 / Steps / steps / 操作步骤 | ❌ | 保留原始编号格式 |
| 预期结果 | 预期结果 / 期望结果 / Expected / expected / 预期 | ❌ | 非编号行自动加 `-` 前缀 |
| 备注 | 备注 / 说明 / Note / note / 注释 | ❌ | 行内展示 |
| 设计方法 | 设计方法 / 测试方法 / Method / method | ❌ | EP/BVA/ST/EG 等 |
| 需求编号 | 需求编号 / 需求ID / ReqID / req_id / 关联需求 | ❌ | 需求追溯标识 |

### 匹配规则

- 大小写不敏感（`"Case ID"` 与 `"case id"` 等效）
- 精确匹配（当前不支持模糊匹配或包含匹配）
- 按别名列表顺序优先匹配（第一个匹配到的生效）
- 一个标准字段最多匹配一个实际列

## 模块名合并规则

Excel 中模块列可能出现以下两种格式：
- 短编号：`F1`、`F2`、`F11`
- 带名称：`F1 平台资源池与租户资源管理`、`F11 模型供应商MODEL_ID变更影响治理`

**处理逻辑**：以空格分隔，取第一段作为模块编号，后面内容忽略（名称由 `module_names` 映射表提供）。

## 预期结果格式化规则（正向）

| 原始文本 | 转换后 |
|---------|--------|
| `1.验证页面展示` | `1.验证页面展示` |
| `10.超长步骤场景` | `10.超长步骤场景` |
| `- 搜索结果为空` | `- 搜索结果为空` |
| `• 数据一致` | `• 数据一致` |
| `* 重点验证` | `* 重点验证` |
| `提示发布成功` | `- 提示发布成功` |

即：已有编号（1-99）或已知列表符号的行保留原格式，纯文本行自动加 `- ` 前缀。

## 反向映射规则（md → xlsx）

### 引用块还原

正向转换将前置条件的每一行加了 `> ` 前缀变为引用块。反向转换时逐行去掉 `> ` 前缀：

| Markdown 中的文本 | 还原后 |
|------------------|--------|
| `> 1.平台管理员已登录;` | `1.平台管理员已登录;` |
| `> 2.模型库中存在已纳管模型M` | `2.模型库中存在已纳管模型M` |
| `>` | （空行，保留为空字符串） |

### 预期结果 `- ` 前缀剥离

正向转换对不含编号或列表符号的纯文本行自动加了 `- ` 前缀。反向转换时逐行识别并去掉：

| Markdown 中的文本 | 还原后 | 说明 |
|------------------|--------|------|
| `1.搜索结果展示CL0100` | `1.搜索结果展示CL0100` | 编号行保持不动 |
| `- 提示发布成功` | `提示发布成功` | 自动添加的 `- ` 被剥离 |
| `- 原本就有横线的数据` | `原本就有横线的数据` | ⚠️ 无法区分是自动添加还是原文自带 |
| `• 页面无异常弹窗` | `• 页面无异常弹窗` | 已知列表符号保持不动 |

### 可选字段缺失处理

如果 Markdown 中不存在某字段的 `**xxx**：` 标记，对应字段值设为空字符串 `""`，写入 xlsx 时该单元格为空。

### 模块列还原

默认模式下模块列还原为短编号（如 `F1`）。若 `config.json` 的 `expand_module_name` 为 `true`，则还原为长形式 `F1 平台资源池与租户资源管理`。

---

## 有损转换说明

预期结果的 `- ` 前缀剥离是唯一的有损场景：若原始 Excel 单元格中有不以编号开头但本身以 `- ` 开头的行（如清单项），反向转换时会误将其视为自动添加的前缀而去掉。此类情况在实际测试用例中极少出现，影响可忽略。

## 最小模板（快速上手）

```json
{
  "column_aliases": {
    "用例编号": ["用例编号"],
    "模块":     ["模块"],
    "用例标题": ["用例标题"],
    "优先级":   ["优先级"]
  },
  "module_names": {},
  "priority_levels": ["P0", "P1", "P2", "P3"]
}
```

最小模板只声明必填字段的精确列名，跳过可选字段。`module_names` 为空时模块概览表只展示编号不展示名称。

## 常见适配场景

### 更换优先级体系

某项目使用 Critical/Major/Minor/Trivial：

```json
{
  "priority_levels": ["Critical", "Major", "Minor", "Trivial"],
  "column_aliases": {
    "优先级": ["Severity", "优先级"]
  }
}
```

### 非标准列名

某项目用"TestCase"做标题、"Steps"做步骤、"Pre-condition"做前置条件：

```json
{
  "column_aliases": {
    "用例标题": ["TestCase", "用例标题"],
    "测试步骤": ["Steps", "测试步骤"],
    "前置条件": ["Pre-condition", "前置条件"]
  }
}
```

### 自定义输出列名

交付 xlsx 需要英文列名：

```json
{
  "export_column_names": ["Case ID", "Module", "Title", "Priority", "Precondition", "Steps", "Expected", "Note", "Method", "Req ID"]
}
```

### 新增模块或更新模块名

```json
{
  "module_names": {
    "F17": "新模块名称",
    "F1": "更新后的模块名"
  }
}
```

# 输出模板与规范

---

## 输出规范详细说明

### 用例标题格式（强制）
`【# 需求编号_模块_功能】场景类型：测试点`

**正确示例：**
- `【# REQ_001_LOGIN】正向：输入正确账号密码登录成功`
- `【# REQ_S2_VENDOR】边界：添加第10个供应商配置成功`

**错误示例：**
- ❌ `登录成功测试`（缺少需求编号前缀）
- ❌ `正确输入账号密码`（场景类型不明确）

### 用例编号规则
`TC_{MODULE}_{NNN}`，如 `TC_LOGIN_001`、`TC_VENDOR_001`

**注意：** 模块缩写使用大写英文，编号使用3位数字，统一用下划线分隔。

### 优先级标注
统一使用 **P0/P1/P2/P3**

| 优先级 | 定义 | 覆盖率要求 |
|--------|------|-----------|
| P0 | 阻塞级，核心功能不可用 | 100% |
| P1 | 严重级，主要功能受影响 | 100% |
| P2 | 一般级，次要功能受影响 | 选择性 |
| P3 | 轻微级，边缘场景 | 选择性 |

### 设计方法标记

| 标记 | 含义 | 适用场景 |
|------|------|---------|
| EP | 等价类划分 | 有明确输入范围/格式要求 |
| BVA | 边界值分析 | 验证边界值处理 |
| ST | 场景法 | 涉及业务流程/多步骤操作 |
| EG | 错误推测法 | 历史缺陷多/高风险模块 |

**标记组合示例：** `EP+BVA`、`EP+ST+EG`

### 前置条件要求
- 必须是可验证的客观事实
- 禁止使用"系统正常"、"环境就绪"等模糊表述
- 每条前置条件应独立可验证

**正确示例：**
```
1. 系统正常运行且可访问登录页面
2. 测试手机号已注册账号（13800138000）
3. 该账号未被封禁
```

**错误示例：**
```
❌ 系统正常
❌ 环境已准备好
```

### 测试步骤要求
- 必须是可执行的原子操作
- 每步只做一个动作
- 步骤编号连续（1、2、3...）

**正确示例：**
```
1. 打开登录页面
2. 在手机号输入框输入11位有效大陆手机号
3. 点击"获取验证码"按钮
```

**错误示例：**
```
❌ 输入手机号并点击获取验证码（两个动作）
❌ 验证登录功能是否正常（模糊表述）
```

### 预期结果要求
- 必须是可验证的
- 禁止使用"正常"、"正确"、"成功"等模糊词
- 应描述具体的系统响应或界面变化

**正确示例：**
```
1. 第3步后：按钮变为倒计时状态（60s），提示"验证码已发送"
2. 第6步后：登录成功，页面跳转至首页，顶部显示用户昵称"测试用户"
```

**错误示例：**
```
❌ 系统正常响应
❌ 登录成功
❌ 功能正常
```

### 输出格式
- 所有输出物必须使用 **Markdown格式**
- 表格使用标准Markdown表格语法
- 代码块使用三个反引号包裹

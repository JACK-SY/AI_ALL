# 核心协作机制（10项）

---

## 机制1：变更合约机制

**作用：** 规范需求变更流程，确保变更可追溯

**发挥时机：** 用户说"修改XX用例"/"需求变更"/"增加XX测试点"时触发

**响应动作：** 立即停止当前操作，输出《变更合约》模板：

```
## 变更合约

变更原因：
影响模块：
需要新增的用例范围：
需要修改的用例范围：
需要删除的用例范围：
不受影响的用例范围：
锁定项（绝对不能修改）：
确认执行：□ 是  □ 否
```

**执行规则：** 只有用户明确回复"确认执行"，才能开始修改。

---

## 机制2：四级风险分级机制

**作用：** 根据P0-P3优先级指导用例覆盖深度

**发挥时机：** 所有用例生成时自动应用

| 级别 | 说明 | 覆盖要求 | 步骤要求 |
|------|------|---------|---------|
| **P0** | 阻塞核心业务流程，影响所有用户 | 必须100%覆盖 | ≥3个测试步骤 |
| **P1** | 重要功能异常，影响大部分用户 | 必须覆盖 | 完整步骤 |
| **P2** | 次要功能异常，影响部分用户 | 尽量覆盖 | 标准步骤 |
| **P3** | 边缘功能异常，影响极小 | 选择性覆盖 | 可简化为1个步骤 |

**执行规则：** AI自动根据风险等级调整用例详细程度，P0用例必须增加异常分支覆盖。

---

## 机制3：渐进式细化机制（外圈→中圈→内圈）

**作用：** 确保阶段顺序，严格禁止跨层推进

**发挥时机：** 任意时刻

```
外圈（需求解读）→ 中圈（测试分析）→ 内圈（用例填充）
```

**执行规则：**
- 外圈（需求解读）未确认，禁止进入中圈（测试分析）
- 中圈（测试分析）未确认，禁止进入内圈（用例填充）
- 每圈必须有用户明确的确认回复才能继续
- 禁止跨层推进，任何理由不接受

---

## 机制4：模糊点主动澄清机制

**作用：** 避免测试意图不明确导致返工

**发挥时机：** 需求中出现"大概"/"可能"/"尽量"/"左右"/"差不多"等模糊表述时

**响应动作：** 立即停止生成，将模糊点加入《待澄清问题清单》，等待用户回复：

```
## 待澄清问题清单

❗ 发现模糊点，暂停推进，等待以下问题澄清：

1. [问题描述]：需求中"[原文]"含义不明确，请明确：[具体问题]
2. ...

所有问题澄清后，请回复"已澄清，继续"以推进流程。
```

**执行规则：** 所有模糊点未澄清前，禁止继续推进任何阶段。

---

## 机制5：历史Bug自动关联机制

**作用：** 利用历史缺陷提升测试覆盖率

**发挥时机：** 阶段2测试分析时，自动检索知识库中的《历史bug库》

**响应动作：**
- 检索本模块及同源模块的历史bug
- 将历史bug转化为对应测试用例
- 所有历史bug关联的用例标记为P1优先级

**执行规则：** 所有历史bug必须至少对应1条测试用例，在测试分析报告中列出《历史同源bug覆盖清单》。

---

## 机制6：测试规范自动应用机制

**作用：** 确保用例符合统一标准

**发挥时机：** 所有用例生成时，自动读取《测试规范档案》

**响应动作：** 自动应用规范中的通用规则，包括但不限于：

### 基础通用规则
- 必填项测空值场景
- 删除操作有二次确认场景
- 超长文本边界测试
- 特殊字符输入测试
- 并发操作测试

### UI交互测试规则（必须覆盖）
- **按钮状态**：启用/禁用/隐藏状态的切换
- **Toast提示**：成功/失败/警告提示的内容验证
- **弹窗控件**：输入框/下拉框/复选框的交互
- **页面跳转**：跳转后数据是否刷新
- **数据刷新**：操作后列表/详情是否即时更新

### 跨模块测试规则（必须覆盖）
- **数据联动**：上游操作后下游数据同步（如供应商修改→接入点联动）
- **状态联动**：状态变更的级联影响（如禁用供应商→调用拦截）
- **删除影响**：删除对象对关联数据的影响（如删除供应商→关联接入点失效）

### 端到端流程规则（必须覆盖）
- **全路径验证**：用户操作从开始到结束的完整路径
- **业务闭环**：操作的最终效果与预期一致
- **异常恢复**：异常中断后系统状态与提示

**执行规则：** 所有规则必须严格执行，无需用户每次说明。阶段3骨架生成时必须确保4种场景类型全部覆盖，阶段4细节填充时必须包含UI交互和跨模块联动细节。

## 机制7：版本快照机制

**作用：** 记录每个阶段的输出物，便于追溯

**发挥时机：** 阶段6完成后

**响应动作：**
- 生成当前阶段输出物的版本快照
- 快照格式：`[阶段名称]-v[版本号]-[日期时间]`
- 保存到知识库《版本快照》中

**执行规则：** 所有版本快照不可删除，可随时回溯。

---

## 机制8：重复用例自动去重机制

**作用：** 避免冗余用例，降低维护成本

**发挥时机：** 用例生成时自动执行

**响应动作：**
- 自动比对已存在的用例
- 删除重复用例（标题和测试步骤完全一致者）
- 合并相似用例（测试意图相同但步骤略有差异者）

**执行规则：**
- 用例标题和测试步骤完全一致的视为重复用例，必须删除
- 合并时保留覆盖更全面的版本，并在备注中注明合并来源

---

## 机制9：评审意见闭环机制

**作用：** 确保评审意见被逐条处理

**发挥时机：** 用户提交评审意见时

**响应动作：** 逐条处理评审意见，输出处理结果：

```
## 评审意见处理记录

| 意见编号 | 原始意见 | 处理方式 | 处理结果 | 影响用例 |
|---------|---------|---------|---------|---------|
| R-001   | [意见内容] | 修改/接受/拒绝+原因 | [处理结果说明] | TC_MODULE_001等 |
```

**执行规则：** 所有评审意见必须有明确的处理结果，未处理的意见必须列出并说明原因。

---

## 机制10：失忆预防机制

**作用：** 新对话开始时快速恢复上下文

**发挥时机：** 每次新对话开始时

**响应动作：** 输出《当前状态摘要》：

```
## 当前状态摘要

当前阶段：[阶段名称 / 尚未开始]
已通过门禁：[门禁一 / 门禁二 / ... / 无]
最新版本：[v版本号 / 无]
已生成用例数：[X条 / 0条]（P0:X条 P1:X条 P2:X条 P3:X条）
待完成事项：[列表，无则填"无"]
待澄清问题：[如有则列出，无则填"无"]

---
请告知您希望继续推进还是重新开始？
```

**执行规则：** 每次新对话开始时必须自动输出，等待用户确认或指示。

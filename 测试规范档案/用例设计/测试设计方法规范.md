## 1. 方法选择指南

| 需求特征 | 推荐方法 |
|---------|---------|
| 有明确输入范围/格式要求 | 等价类划分 + 边界值分析 |
| 涉及业务流程/多步骤操作 | 场景法 |
| 历史缺陷多/高风险模块 | 错误推测法 |
| 复杂输入组合 | 等价类 + 场景法 |

## 2. 等价类划分法 (Equivalence Partitioning, EP)

### 2.1 定义

将输入数据划分为若干等价类，从每个等价类中选取代表性数据作为测试数据。同一等价类中的数据对揭露缺陷的效果是等价的。

### 2.2 分类

- **有效等价类**：符合规格说明的合理输入，程序应正确处理
- **无效等价类**：不符合规格说明的非法输入，程序应拒绝并给出提示

### 2.3 划分规则

| 输入条件 | 有效等价类 | 无效等价类 |
|---------|-----------|-----------|
| 取值范围 [a, b] | [a, b] 内的值 | <a 的值, >b 的值 |
| 取值集合 {A, B, C} | A, B, C 各取一个 | 非集合内的值 |
| 布尔值 | true, false | 非布尔类型值 |
| 格式要求（如手机号11位） | 符合格式的值 | 不符合格式的值 |
| 必填项 | 非空有效值 | 空值、null、undefined |
| 长度限制 [m, n] | 长度在 [m, n] 内 | 长度 <m 或 >n |

### 2.4 用例生成原则

1. 一条用例尽可能覆盖多个**有效**等价类（减少用例数）
2. 每个**无效**等价类必须**单独**一条用例覆盖（确保定位问题）

### 2.5 识别模式

从需求中识别以下关键词来划分等价类：

| 模式类型 | 关键词/句式 | 等价类划分 |
|---------|------------|-----------|
| 范围限制 | `在...之间`、`范围`、`不超过`、`至少`、`最多`、`N-M` | 有效区间 / 小于下限 / 大于上限 |
| 集合限制 | `仅限`、`只能是`、`可选值包括`、`支持的类型` | 集合内各值 / 集合外值 |
| 格式限制 | `格式`、`符合`、`正则`、`必须为`、`...格式` | 符合格式 / 不符合格式 |
| 必填标记 | `必填`、`必须`、`不能为空`、`required`、`*` | 有效值 / 空值 |
| 长度限制 | `长度`、`字符数`、`位数`、`最短`、`最长` | 有效长度 / 太短 / 太长 |
| 类型限制 | `数字`、`字母`、`中文`、`整数`、`小数` | 正确类型 / 错误类型 |

### 2.6 示例：用户年龄输入（18-60岁整数）

| 等价类ID | 等价类 | 类型 | 测试数据 | 预期结果 |
|---------|-------|------|---------|---------|
| EC_01 | [18, 60] | 有效 | 25 | 通过 |
| EC_02 | <18 | 无效 | 10 | 提示年龄不符合要求 |
| EC_03 | >60 | 无效 | 70 | 提示年龄不符合要求 |
| EC_04 | 非数字 | 无效 | "abc" | 提示格式错误 |
| EC_05 | 空值 | 无效 | null | 提示必填 |
| EC_06 | 小数 | 无效 | 25.5 | 提示需为整数 |

## 3. 边界值分析法 (Boundary Value Analysis, BVA)

### 3.1 定义

对输入/输出的边界进行测试。经验表明，大量缺陷发生在边界附近。

### 3.2 边界点定义

- **上点 (On Point)**：边界上的点
- **离点 (Off Point)**：离边界最近的点（边界外侧）
- **内点 (In Point)**：域内任意一点

### 3.3 取值规则

| 边界类型 | 测试点 | 示例（范围 1-100） |
|---------|--------|------------------|
| [a, b] 闭区间 | a-1, a, b, b+1 | 0, 1, 100, 101 |
| (a, b) 开区间 | a, a+1, b-1, b | 1, 2, 99, 100 |
| [a, b) 左闭右开 | a-1, a, b-1, b | 0, 1, 99, 100 |
| (a, b] 左开右闭 | a, a+1, b, b+1 | 1, 2, 100, 101 |
| 最大长度 N | N-1, N, N+1 | - |
| 最小长度 M | M-1, M, M+1 | - |

### 3.4 与等价类结合使用

标准流程：
1. 先用等价类划分确定输入域
2. 对每个等价类的边界进行边界值分析
3. 边界值用例独立于等价类用例

### 3.5 识别模式

从需求中识别以下关键词提取边界：

| 边界类型 | 关键词 | 提取方法 |
|---------|--------|---------|
| 数值边界 | `最小`、`最大`、`不超过`、`至少`、`≤`、`≥`、`<`、`>` | 提取数值N，测试N-1, N, N+1 |
| 长度边界 | `长度`、`字符数`、`位数`、`最短M`、`最长N` | 测试M-1, M, N, N+1 |
| 数量边界 | `最多N个`、`至少M条`、`限制N次` | 测试M-1, M, N, N+1 |
| 时间边界 | `N分钟内`、`超过N小时`、`等待N秒` | 测试N-1, N, N+1 (时间单位) |
| 金额边界 | `最低N元`、`上限M`、`满N减` | 测试N-0.01, N, M, M+0.01 |

### 3.6 示例：密码长度6-20位

| 测试点 | 类型 | 值 | 预期结果 |
|-------|------|-----|---------|
| 下边界-1 | 离点 | 5位 | 失败：密码太短 |
| 下边界 | 上点 | 6位 | 成功 |
| 上边界 | 上点 | 20位 | 成功 |
| 上边界+1 | 离点 | 21位 | 失败：密码太长 |
| 内点 | 内点 | 12位 | 成功 |

## 4. 场景法 (Scenario Testing, ST)

### 4.1 定义

通过模拟用户实际使用场景，覆盖业务流程中的各种路径。适用于事件触发型业务流程。

### 4.2 流程类型

| 流程类型 | 定义 | 优先级 |
|---------|------|--------|
| **基本流** | 最常见的正常操作路径（主成功场景） | P0 |
| **备选流** | 可接受的替代路径，最终仍能完成目标 | P1 |
| **异常流** | 错误处理路径，触发错误提示或回退 | P2 |

### 4.3 场景识别方法

1. **从需求中提取**：
   - 用户故事格式：`作为...我想要...以便于...`
   - 前置条件、后置条件
   - 流程步骤描述

2. **识别条件分支**：
   - `当...时`、`如果...则`、`否则`
   - `或者`、`也可以`
   - `失败时`、`超时后`、`错误情况`

3. **流程图梳理**：
   - 绘制流程图标记所有分支
   - 每条路径生成一个场景

### 4.4 用例生成规则

| 场景类型 | 用例覆盖要求 |
|---------|-------------|
| 基本流 | 至少1条用例完整覆盖整个流程 |
| 备选流 | 基本流 + 该备选流分支 |
| 异常流 | 触发点 + 异常处理验证 |

### 4.5 流程类型识别关键词

| 流程类型 | 识别关键词/句式 |
|---------|----------------|
| 基本流 | `正常情况`、`成功后`、主语句（无条件分支的陈述句） |
| 备选流 | `也可以`、`或者`、`另外支持`、`如果选择` |
| 异常流 | `如果失败`、`当...无效时`、`错误情况`、`超时`、`异常`、`否则` |

## 5. 错误推测法 (Error Guessing, EG)

### 5.1 定义

基于测试经验和直觉，推测程序可能存在的缺陷位置，有针对性地设计测试用例。这是对其他方法的补充。

### 5.2 输入相关错误推测

| 错误类型 | 测试数据示例 |
|---------|-------------|
| 空值 | `null`、`undefined`、`""`、空格 |
| 特殊字符 | `' " < > & \ / # @ ! $ % ^ * ( )` |
| 极端值 | `0`、`-1`、`MAX_INT`、超长字符串（1000+字符） |
| 格式错误 | 错误日期格式、无效邮箱、短于11位手机号 |
| 注入攻击 | `' OR 1=1 --`（SQL）、`<script>alert(1)</script>`（XSS） |
| Unicode | 中文、日文、emoji、特殊Unicode字符 |

### 5.3 操作相关错误推测

| 错误类型 | 测试场景 |
|---------|---------|
| 重复提交 | 快速双击提交按钮 |
| 并发操作 | 多窗口/多Tab同时操作同一数据 |
| 操作中断 | 提交过程中刷新页面、关闭浏览器 |
| 顺序颠倒 | 跳过必要步骤直接访问后续页面 |
| 回退操作 | 提交后点击浏览器返回按钮 |

### 5.4 数据相关错误推测

| 错误类型 | 测试场景 |
|---------|---------|
| 边界数据 | 第一条记录、最后一条记录 |
| 重复数据 | 相同内容重复提交 |
| 关联数据 | 删除被其他数据引用的记录 |
| 历史数据 | 过期数据的显示与处理 |
| 空列表 | 无数据时的页面展示 |

### 5.5 环境相关错误推测

| 错误类型 | 测试场景 |
|---------|---------|
| 浏览器兼容 | Chrome、Firefox、Safari、Edge |
| 屏幕适配 | 移动端、平板、PC各分辨率 |
| 弱网环境 | 2G/3G网络、高延迟、丢包 |
| 时区差异 | 不同时区的时间显示 |

### 5.6 自动触发规则

当需求中出现以下元素时，自动添加错误推测用例：

| 触发元素 | 推测错误类型 | 生成用例 |
|---------|-------------|---------|
| 输入框 | 特殊字符、超长输入、空值 | SQL注入、XSS、空格处理 |
| 提交按钮 | 重复提交、并发 | 快速双击、多窗口操作 |
| 文件上传 | 格式、大小、内容 | 非法格式、超大文件、空文件 |
| 列表/表格 | 边界记录 | 首条、末条、空列表 |
| 搜索功能 | 特殊查询 | 通配符、无结果、大结果集 |
| 金额字段 | 精度问题 | 小数点、负数、0元 |
| 日期时间 | 格式和边界 | 跨天、跨年、时区 |
| 表单提交 | 操作中断 | 网络断开、页面刷新 |

### 5.7 输入类型→错误推测映射

| 输入类型 | 推测的异常输入 |
|---------|---------------|
| 文本框 | `' " < > & \ / # @ ! $ %`、空格、超长、null |
| 数值框 | 负数、0、MAX_INT、小数、非数字字符 |
| 手机号 | 非1开头、非11位、非数字、空号段 |
| 邮箱 | 缺少@、多个@、特殊域名、超长 |
| 密码 | 纯数字、纯字母、含空格、常见弱密码 |
| 日期 | 过去日期、未来日期、无效日期(2/30) |
| URL | 非http(s)、恶意链接、过长URL |

### 5.8 模块类型→错误推测映射

| 模块类型 | 推测的错误场景 |
|---------|---------------|
| 登录 | 账号不存在、密码错误、账号锁定、会话过期 |
| 注册 | 重复注册、验证码错误、协议未勾选 |
| 支付 | 余额不足、支付超时、重复支付、金额为0 |
| 订单 | 库存不足、并发下单、订单取消后操作 |
| 文件 | 上传失败、下载中断、格式不支持 |
| 权限 | 越权访问、Token过期、角色切换 |
| 模型部署 | 镜像缺失、端口冲突、资源不足、并发部署 |
| API接口 | 参数缺失、参数超限、鉴权失败、限流触发 |

## 6. 方法组合策略

### 6.1 覆盖优先级映射

| 优先级 | 覆盖内容 |
|--------|---------|
| **P0** | 基本流用例 + 核心有效等价类 |
| **P1** | 备选流用例 + 边界值 + 主要无效等价类 |
| **P2** | 异常流用例 + 次要无效等价类 + 常见错误推测 |
| **P3** | 边缘错误推测 + 环境兼容性 |

### 6.2 设计方法标记

| 标记 | 全称 |
|------|------|

| BVA | Boundary Value Analysis（边界值分析） |
| ST | Scenario Testing（场景法） |
| EG | Error Guessing（错误推测） |
| EP+BVA | 组合使用 |

## 7. 测试视角覆盖规范

### 7.1 问题背景

实际测试中发现以下类型用例容易遗漏：
- **权限越权**：低权限用户通过API/URL越权访问高权限功能
- **多租户隔离**：同名资源在不同租户下策略是否串扰
- **计费排除项**：哪些调用不计入限额（如租户级Key）
- **状态交叉**：资源处于非正常状态（禁用/离职/过期）时关联功能行为

### 7.2 三层分析法（强制执行）

用例生成时必须按以下三层逐层分析，不可只做第一层：

| 层级 | 分析内容 | 覆盖方法 |
|------|---------|---------|
| **第一层：PRD正向流程** | 功能描述、验收标准对应的正常路径 | EP（有效等价类）+ ST（基本流） |
| **第二层：异常路径与边界** | 输入边界、状态转换、权限不足、并发竞态 | BVA + ST（异常流）+ EG |
| **第三层：架构与安全视角** | 多租户隔离、越权访问、计费排除、跨模块状态交叉 | 见7.3测试视角清单 |

### 7.3 测试视角清单（用例生成前强制过一遍）

每次生成用例前，必须对照以下清单逐项检查是否有对应用例：

| # | 测试视角 | 检查问题 | 适用场景 | 对应设计方法 |
|---|---------|---------|---------|-------------|
| 1 | **权限边界** | 低权限用户能否通过API/URL越权访问？租户管理员能否操作平台级功能？ | 所有涉及权限管控的功能 | EG |
| 2 | **多租户隔离** | 同名资源在不同租户下是否隔离？策略配置是否串扰？ | 多租户SaaS平台 | EG |
| 3 | **计费/限额排除项** | 哪些调用不计入限额？为什么？（如系统Key、租户级Key） | 用量统计、限额管理 | EG |
| 4 | **状态交叉** | 资源处于非正常状态（禁用/离职/过期）时，关联功能行为如何？ | 所有有生命周期状态的资源 | ST（异常流）|
| 5 | **数据一致性** | 多处展示同一数据时是否一致？修改后是否同步？ | 跨页面展示同一数据 | ST（备选流）|
| 6 | **并发竞态** | 同一资源并发操作时是否有冲突？自动+手动操作是否冲突？ | 状态流转、批量操作 | EG |
| 7 | **降级容灾** | 依赖服务不可用时行为如何？是否有兜底？ | 依赖外部服务的功能 | ST（异常流）|
| 8 | **历史数据兼容** | 升级后历史数据是否正常展示和使用？ | 版本迭代、数据迁移 | EG |

### 7.4 跨模块关联矩阵（适用于跨功能模块分析）

当功能涉及多个模块交叉时，必须绘制资源状态×关联功能矩阵，识别需要验证的交叉点：

**矩阵模板：**

| 资源状态 \ 关联功能 | 功能A | 功能B | 功能C |
|-------------------|-------|-------|-------|
| **正常状态** | ✅正常 | ✅正常 | ✅正常 |
| **禁用状态** | ❓验证 | ❓验证 | ❓验证 |
| **过期状态** | ❓验证 | ❓验证 | ❓验证 |
| **异常状态** | ❓验证 | ❓验证 | ❓验证 |

**示例（API Key状态×关联功能）：**

| Key状态 \ 关联功能 | 深链生成 | 实际调用 | 限额统计 | 护栏检测 |
|------------------|--------|---------|---------|--------|
| 正常Key | ✅正常 | ✅正常 | ✅计入 | ✅执行 |
| 禁用Key | 需验证 | 需验证 | 需验证 | 需验证 |
| 租户级Key | ✅正常 | ✅正常 | 需验证 | ✅执行 |
| 离职Key | 需验证 | 需验证 | 需验证 | 需验证 |

> 标记为"需验证"的交叉点即为需要补充的测试用例。

### 7.6 异常场景测试

#### 7.6.1 输入异常

| 测试点 | 描述 | 示例 | 对应设计方法 |
|-------|------|------|-------------|
| 非法输入 | 特殊字符、空值、超出范围的值 | 输入 `' OR 1=1 --`、空字符串、负数 | EG |
| 格式错误 | 数据格式不符合要求 | 手机号输入字母、邮箱缺少 `@` | EP |
| 必填字段 | 未填写必填项时的处理 | 发布智能体时场景分类未填写 | EP |
| 类型错误 | 输入类型与字段类型不匹配 | 数字字段输入字符串 | EP |

#### 7.6.2 流程异常

| 测试点 | 描述 | 示例 | 对应设计方法 |

## 8. 用例防重复与质量控制规范

### 8.1 问题识别清单

| # | 问题类型 | 识别特征 | 示例 |
|---|---------|---------|------|
| 1 | **功能已取消** | 用例中出现PRD已明确不做的功能描述 | "发布向导"、"上下架"等已移除功能 |
| 2 | **用例内容重复** | 两条或多条用例的测试步骤和预期结果基本相同 | 撤销授权和黑名单封禁的验证点完全一致 |
| 3 | **端到端用例与单功能用例重复** | 端到端用例包含了其他单功能用例的完整验证 | TC_RES_030包含了TC_RES_003的所有步骤 |
| 4 | **编号不连续** | 用例编号存在大量空缺 | TC_RES_001后直接到TC_RES_003 |
| 5 | **用例粒度不当** | 单条用例步骤超过8步或预期结果超过6项 | 端到端用例14步11项预期结果 |

### 8.2 重复用例判定规则

**判定条件（满足任一即为重复）：**

| 判定类型 | 条件 | 处理方式 |
|---------|------|---------|
| **内容完全重复** | 测试步骤和预期结果相似度 ≥ 80% | 删除重复用例，保留优先级高的 |
| **验证点重复** | 核心验证点相同，仅前置条件或数据不同 | 合并为一条用例，参数化数据 |
| **端到端包含** | 端到端用例完整包含单功能用例的验证点 | 端到端用例简化为流程验证，详细验证留在单功能用例 |
| **不同模块重复** | 跨模块用例验证相同业务规则 | 保留一个模块的用例，其他模块标注引用 |

**相似度计算方法：**
- 将测试步骤和预期结果拆分为独立句子
- 比较相同句子数量占总句子数的比例
- ≥80%判定为重复

### 8.3 防重复检查流程（用例生成后强制执行）

```
生成用例 → 检查1: 功能取消项过滤 → 检查2: 后端操作前端化检查 → 检查3: 重复用例检测 → 检查4: 编号连续性检查 → 检查5: 粒度检查 → 检查6: 数据保留检查 → 输出最终用例
```

#### 检查1：功能取消项过滤

- **输入来源**：需求文档分拆目录下 `00_文档头与通用章节.md` 的 **§4a 移出本期内容清单**
- **规则**：用例标题、步骤、预期中出现取消功能关键词的用例必须删除
- **关键词示例**：发布向导、上下架、负载均衡、随机、轮询、健康降级、复杂度路由等已移除功能名称

#### 检查2：后端操作前端化检查

- **规则**：所有测试用例必须从前端可验证的视角设计，禁止编写仅包含后端操作（如ES索引操作、数据库查询、API接口直接调用等）的用例
- **例外情况**：仅当功能明确为纯后端功能（无前端页面、无用户交互、纯定时任务或系统后台处理）时，允许编写后端视角的用例
- **判断标准**：
  - ❌ 禁止："检查ES索引"、"查询数据库表"、"验证Redis缓存"、"检查写别名指向"等
  - ✅ 允许："查看调用日志详情页"、"点击查看原文按钮"、"验证页面提示文案"等
- **转换方法**：将后端操作转换为前端可观察的表现，如：
  - "ES索引过期" → "页面展示'原文已超过保留期限'提示"
  - "写别名滚动" → "跨月查询数据正常展示"
  - "内容写入失败" → "页面不展示查看原文按钮，显示写入失败提示"

#### 检查3：重复用例检测

| 检查项 | 方法 | 工具 |
|-------|------|------|
| 标题重复检测 | 检查用例标题相似度 | 字符串相似度算法 |
| 步骤重复检测 | 检查测试步骤相似度 | 句子级比较 |
| 验证点重复检测 | 提取每个用例的核心验证点，检查重复 | 验证点关键词提取 |

#### 检查4：编号连续性检查

- **规则**：同一子模块内用例编号必须连续
- **处理**：编号空缺时重新编排，删除的用例编号不再复用

#### 检查5：粒度检查

| 检查项 | 阈值 | 处理方式 |
|-------|------|---------|
| 测试步骤数 | >8步 | 拆分为多条用例 |
| 预期结果数 | >6项 | 拆分为多条用例或合并同类项 |
| 用例复杂度 | 涉及多个独立功能点 | 拆分为单功能用例 |

#### 检查6：数据保留检查

- **目的**：确保涉及删除操作的场景，验证历史数据仍保留可查询
- **规则**：当功能涉及删除操作（软删除或硬删除）时，必须验证删除后历史数据是否仍可查询
- **适用范围**：
  - 资源删除后历史调用记录是否保留（如API Key删除后，用量统计仍可查询）
  - 配置删除后历史数据是否保留（如限额配置删除后，历史用量账本是否保留）
  - 实体删除后关联数据是否保留（如接入点删除后，历史调用日志是否保留）
- **检查方法**：
  1. 识别所有涉及删除操作的用例
  2. 检查是否有对应用例验证"删除后历史数据仍可查询"
  3. 如缺失，补充"删除后历史数据保留"验证用例
- **验证要点**：
  - 删除后历史数据是否仍可在查询页面搜索到
  - 删除后历史数据的完整性和准确性
  - 删除后数据保留期限是否符合需求（如保留12个月）
  - 跨模块数据一致性：A模块删除的数据，B模块查询是否正常
- **常见遗漏场景**：
  - 离职人员API Key删除后，用量统计和调用日志查询
  - 限额配置删除后，历史用量账本查询
  - 接入点删除后，历史调用记录查询
  - 模型下线后，历史调用记录查询

#### 8.3.1 不做功能记录规范

| 规范项 | 规则 | 说明 |
|-------|------|------|
| **记录位置** | 仅在 `00_文档头与通用章节.md` 的 **§4a 移出本期内容清单** 中记录 | 功能模块文件（01_F1_xxx.md 等）中不得包含不做功能的描述 |
| **记录格式** | 表格形式，包含：移出内容、原编号/来源、Backlog归属、移出时间/决策 | 见 §4a 表格格式 |
| **功能模块文件** | 不得提及不做的功能名称 | 避免Agent读取时产生脏数据 |
| **验收标准** | 不得包含验证"不做功能无入口"的验收项 | 如"健康降级无配置入口"不应作为验收标准 |

#### 8.3.2 质量过滤检查位置

| 检查项 | 检查位置 | 检查方法 |
|-------|---------|---------|
| **已取消功能** | `00_文档头与通用章节.md` §4a | 对比用例中是否出现§4a记录的功能关键词 |
| **后端操作前端化** | 当前生成的用例文件 | 检查用例步骤/预期中是否包含后端操作描述 |
| **重复用例** | 当前生成的用例文件 | 按8.2判定规则检查内容相似度 |
| **编号连续性** | 当前生成的用例文件 | 检查同一前缀下编号是否连续 |
| **用例粒度** | 当前生成的用例文件 | 检查步骤数≤8、预期结果数≤6 |
| **需求追溯** | 当前生成的用例文件 + 需求追溯矩阵 | 检查每个用例是否关联有效需求编号 |
| **数据保留** | 当前生成的用例文件 + 需求文档 | 检查涉及删除操作的场景是否有"删除后历史数据保留"验证用例 |

### 8.4 端到端用例与单功能用例的职责划分

| 用例类型 | 职责范围 | 粒度要求 | 与其他用例关系 |
|---------|---------|---------|---------------|
| **单功能用例** | 验证单个功能点的所有场景（正常、异常、边界） | 细粒度，步骤≤5步 | 独立存在，不被其他用例替代 |
| **端到端用例** | 验证完整业务流程的连通性和数据流转 | 粗粒度，步骤≤10步 | 只验证流程打通，不重复验证单功能细节 |

**端到端用例简化规则：**
- 不重复验证单功能用例已覆盖的详细场景
- 每个功能点仅验证"成功"状态，异常场景留给单功能用例
- 重点验证跨模块数据流转的正确性

### 8.5 编号管理规范

| 规则 | 说明 |
|------|------|
| **子模块前缀** | 同一需求下不同子模块使用不同前缀（如TC_RES、TC_AUTH） |
| **连续性** | 同一前缀下编号必须连续，不能跳过 |
| **重新编排** | 删除用例后，后续编号重新编排，不保留空缺 |
| **版本管理** | 用例变更时在标题中标记版本号（如v2），不修改历史编号 |

### 8.6 执行规则

1. **用例生成后**：必须按8.3流程执行所有检查
2. **检查1失败**：删除包含取消功能的用例，记录删除原因
3. **检查2失败**：将后端操作转换为前端可验证的用例，重新设计步骤和预期
4. **检查3失败**：合并或删除重复用例，保留优先级高的
5. **检查4失败**：重新编排编号，确保连续性
6. **检查5失败**：拆分用例，确保粒度合理
7. **检查6失败**：补充"删除后历史数据保留"验证用例，确保跨模块数据一致性
8. **检查结果记录**：所有检查结果记录在测试分析报告中

## 9. 用例格式规范

### 9.1 Markdown用例模板

每条测试用例必须遵循以下格式：

```markdown
### TC_{模块}_{序号}

标题：【#{需求编号}_{子模块}_{功能点}】场景类型：测试点

优先级：P0/P1/P2/P3

前置条件：
> 1. 条件1
> 2. 条件2

测试步骤：
1. 步骤1
2. 步骤2

预期结果：
1. 结果1
2. 结果2

设计方法：EP/BVA/ST/EG（可组合如 EP+BVA）

需求追溯：#{需求编号}

***
```

### 9.2 格式规则

| 规则项 | 要求 | 说明 |
|-------|------|------|
| **字段位置** | 字段名与内容必须在同一行 | 如 `标题：内容`，不能分成两行 |
| **分隔符** | 用例之间用 `***` 分隔 | 空一行后使用 `***` |
| **编号格式** | `TC_{模块}_{序号}`，序号3位补零 | 如 TC_RES_001、TC_AUTH_002 |
| **标题格式** | `【#{需求编号}_{子模块}_{功能点}】场景类型：测试点` | 严格遵守此格式 |
| **场景类型** | 正向、异常、边界、交互、反向 | 根据实际场景选择 |
| **优先级** | P0/P1/P2/P3 | 按6.1覆盖优先级映射确定 |
| **前置条件** | 每行以 `>` 开头 | Markdown引用格式 |
| **测试步骤** | 每行以数字开头，格式为 `数字. 内容` | 如 `1. 进入运营管理页面` |
| **预期结果** | 每行以数字开头，格式为 `数字. 内容` | 如 `1. 页面加载成功` |
| **设计方法** | 建议标记，可组合 | 如 `EP+BVA`、`ST+EG`；过渡期可暂不填写 |
| **需求追溯** | 建议关联需求编号 | 如 `#F1_RES_001`；过渡期可暂不填写 |

### 9.3 用例标题命名规范

**格式：** `【#{需求编号}_{子模块}_{功能点}】场景类型：测试点`

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| 需求编号 | 项目级唯一编号 | F1_RES |
| 子模块 | 功能子模块名称 | 授权范围 |
| 功能点 | 具体功能描述 | 资源授权范围配置 |
| 场景类型 | 测试场景类型 | 正向、异常、边界、交互、反向 |
| 测试点 | 具体测试内容 | 配置模型授权给租户 |

**示例：** `【#F1_RES_授权范围】正向：资源授权范围页配置模型授权给租户`

### 9.4 质量指标

| 指标 | 阈值 | 说明 |
|------|------|------|
| 测试步骤数 | ≤8步 | 超过时拆分为多条用例 |

| 前置条件数 | ≤5条 | 超过时简化或拆分为多条用例 |

## 10. 需求追溯矩阵模板

### 10.1 追溯矩阵格式

| 需求编号 | 需求描述 | 用例ID | 用例标题 | 优先级 | 设计方法 | 覆盖状态 |
|---------|---------|--------|---------|--------|---------|---------|
| #F1_RES_001 | 资源授权范围配置 | TC_RES_001 | 资源授权范围页配置模型授权给租户 | P0 | EP+ST | ✅ |
| #F1_RES_002 | 黑名单封禁 | TC_RES_005 | 黑名单拒绝与未授权对外同码 | P1 | EG | ✅ |

### 10.2 覆盖状态说明

| 状态标记 | 含义 | 说明 |
|---------|------|------|
| ✅ | 已覆盖 | 该需求已有对应测试用例 |
| ❓ | 待确认 | 该需求是否需要测试用例待确认 |
| ⏳ | 进行中 | 用例正在设计中 |
| ❌ | 不覆盖 | 该需求无需测试用例（需标注原因） |

### 10.3 需求状态管理

当PRD中需求状态发生变更时，必须更新追溯矩阵：

| 需求状态变更 | 处理方式 | 示例 |
|------------|---------|------|
| **功能取消** | 将对应行标记为❌，标注"功能已取消" | #F4_PUBLISH_001：发布向导已移除 |
| **需求变更** | 更新用例标题和内容，保留原编号 | 测试点从A变更为A+B |
| **新增需求** | 在矩阵中新增行 | #F6_QUO_001：用户级Token限额 |

### 10.4 追溯矩阵维护规则

1. **用例生成时**：同步更新追溯矩阵，确保每个需求至少关联一条用例
2. **需求变更时**：立即更新追溯矩阵中对应行的状态和用例关联
3. **用例删除时**：检查追溯矩阵，确保关联需求仍有其他用例覆盖
4. **测试执行后**：在矩阵中记录用例执行结果（通过/失败/阻塞）
5. **版本快照时**：导出当前追溯矩阵作为版本记录的一部分

# AI模型平台 — 测试规范总纲

> 版本：v1.1 | 生效日期：2026-07-17 | 适用项目：AI模型平台
> **项目级强制规范**：本规范档案为AI模型平台项目测试工作的统一标准，适用于项目内所有测试活动和自动化流程。

---

## 1. 规范概述

本规范档案定义了AI模型平台项目测试工作的统一标准，涵盖测试设计方法、优先级分类、需求追溯、解析规则、交互流程等维度，确保测试产出专业、可追溯、可持续改进。

## 2. 核心约束

### 2.1 用例标题格式（强制）

所有用例标题统一采用以下格式，不得偏离：

```
【# 需求编号_模块_功能】场景类型：测试点
```

| 要素 | 说明 | 示例 |
|------|------|------|
| 需求编号 | 来源需求ID（同一需求所有用例使用相同编号） | REQ_001、REQ-123、MOD_LOGIN_001 |
| 模块 | 功能模块名称或缩写 | LOGIN、CART、MODEL |
| 功能 | 具体功能点 | 账号密码登录、添加商品 |
| 场景类型 | 正向/异常/边界/交互 | 正向、异常、边界、交互 |
| 测试点 | 具体测试内容描述 | 输入正确账号密码登录成功 |

**场景类型枚举**（与SY-generate-test-cases技能阶段3场景类型一致）：

| 场景类型 | 适用情况 | 对应设计方法 |
|---------|---------|-------------|
| 正向 | 验证正常输入/操作的正确行为、核心业务流程 | EP(有效等价类)、ST(基本流) |
| 异常 | 验证异常输入/错误操作/系统异常的处理 | EP(无效等价类)、ST(异常流)、EG(错误推测) |
| 边界 | 验证边界值的处理 | BVA(边界值分析) |
| 交互 | 验证UI控件状态变化、toast提示、弹窗交互、页面跳转 | ST(备选流)、UI交互测试 |

> **注意**：原"反向"场景类型已合并至"异常"场景类型中（反向验证的异常输入/错误操作属于异常场景范畴），新增"交互"场景类型覆盖UI交互维度。

### 2.2 编号一致性（强制）

同一需求下的所有用例，编号前缀（需求编号_模块_功能）保持一致，仅场景类型和测试点不同。

### 2.3 需求追溯（强制）

所有用例必须关联需求ID，不得出现无来源的孤儿用例。

### 2.4 方法标记（强制）

所有用例必须标记设计方法：EP、BVA、ST、EG，或组合标记如 EP+BVA。

### 2.5 优先级自动判定与统计

> 详细规则请参阅《优先级与回归分类规范》第6-7节

优先级由设计方法类型 + 风险因子矩阵自动推导，不对各优先级占比设目标比例。最终分布仅做事后统计和健康检查。

| 优先级 | 定义 | 执行时机 |
|--------|------|---------|
| P0 | 阻塞级，核心功能不可用 | 冒烟测试 |
| P1 | 严重级，影响核心业务流程 | 核心回归 |
| P2 | 一般级，次要功能或异常处理 | 全量回归 |
| P3 | 轻微级，边缘场景 | 有空执行 |

**失衡预警**（事后检查信号，非前置约束）：
- P0占比 >25% → 过度标记，需重新评估
- P0占比 <5% → 可能遗漏关键场景
- P3占比 >30% → 可能过度测试边缘场景

## 3. 用例设计标准流程

```
1. 等价类划分 → 识别所有输入的有效/无效等价类
       ↓
2. 边界值分析 → 对每个等价类的边界进行测试
       ↓
3. 场景法     → 覆盖主要业务流程（基本流+备选流+异常流）
       ↓
4. 错误推测   → 补充经验性测试用例
```

方法选择指南：

| 需求特征 | 推荐方法 |
|---------|---------|
| 有明确输入范围/格式要求 | 等价类划分 + 边界值分析 |
| 涉及业务流程/多步骤操作 | 场景法 |
| 历史缺陷多/高风险模块 | 错误推测法 |
| 复杂输入组合 | 等价类 + 场景法 |

## 4. 优先级与回归分类

### 4.1 优先级定义

| 级别 | 名称 | 定义 | 执行时机 |
|------|------|------|---------|
| P0 | 阻塞级 | 核心功能，不可用则系统无法使用 | 冒烟测试 |
| P1 | 严重级 | 主要功能，影响核心业务流程 | 每日回归 |
| P2 | 一般级 | 次要功能，不影响主流程 | 全量回归 |
| P3 | 轻微级 | 体验优化，边缘场景 | 有空执行 |

### 4.2 回归测试集

| 测试集 | 包含用例 | 执行频率 | 预期耗时 |
|--------|---------|---------|---------|
| 冒烟测试集 | 所有P0用例 | 每次构建 | <30分钟 |
| 核心回归集 | P0+P1用例 | 每日/每次提测 | 2-4小时 |
| 全量回归集 | 所有用例 | 发版前 | 1-2天 |

## 5. 需求追溯

### 5.1 双向追溯

- **正向追溯**：需求ID → 关联用例列表
- **反向追溯**：用例 → 来源需求

### 5.2 覆盖率标准

- 需求覆盖率目标：≥95%
- 核心需求覆盖深度：≥5用例/需求
- 一般功能覆盖深度：3-5用例/需求
- 边缘功能覆盖深度：1-3用例/需求

## 6. 测试产出物标准

### 6.1 测试用例产出物

**主要产出物为Markdown格式文件**（与SY-generate-test-cases技能流水线一致），阶段6额外导出CSV格式用于导入测试管理工具。

**Markdown标准列定义**：

| 列名 | 说明 | 数据类型 |
|------|------|---------|
| 用例编号 | TC_{MODULE}_{SEQ:03d} | 文本 |
| 模块名称 | 功能模块 | 文本 |
| 用例标题 | 【# 需求编号_模块_功能】场景类型：测试点 | 文本 |
| 优先级 | P0/P1/P2/P3 | 文本 |
| 关联需求ID | REQ_XXX（支持多个，逗号分隔） | 文本 |
| 设计方法 | EP/BVA/ST/EG | 文本 |
| 前置条件 | 执行前提（必须可验证，禁止模糊表述） | 文本 |
| 测试步骤 | 详细操作步骤（原子操作，每步只做一个动作） | 文本 |
| 预期结果 | 期望行为（禁止"正常"/"正确"等模糊词） | 文本 |
| 测试类型 | 功能/接口/UI交互/数据联动/异常场景/端到端/安全 | 文本 |

**CSV导出格式**（阶段6用于导入禅道/Jira）：

```csv
用例编号,模块,用例标题,优先级,前置条件,测试步骤,预期结果,测试类型,关联需求ID,设计方法
```

**额外章节**（启用追溯时）：
- **需求追溯矩阵**：需求ID、关联用例、覆盖状态（正向追溯+反向追溯）
- **覆盖率统计**：总需求数、覆盖率、优先级统计

### 6.2 测试计划Markdown

固定8章节结构：
1. 测试目标
2. 测试范围
3. 测试策略
4. 测试环境
5. 测试资源
6. 测试进度
7. 风险与应对
8. 交付物

### 6.3 测试报告Markdown

固定5章节结构：
1. 执行概述
2. 测试统计
3. 缺陷分析
4. 覆盖率报告
5. 结论与建议

## 8. 验收标准

测试工作验收需满足以下条件：

- [ ] 每个需求至少有1条正向用例
- [ ] 每个需求的验收条件都有对应用例
- [ ] 核心需求(P0)覆盖深度 ≥5
- [ ] 无孤儿用例
- [ ] 需求覆盖率 ≥95%
- [ ] 高风险模块有错误推测用例补充
- [ ] 优先级由自动判断逻辑自然产生，仅做事后统计，失衡信号触发重新评估
- [ ] 所有用例标题格式统一
- [ ] 所有用例标记了设计方法
- [ ] 所有用例关联了需求ID
- [ ] **测试视角清单全覆盖**：权限边界、多租户隔离、计费排除项、状态交叉等视角均有对应用例或标注排除原因（详见《测试设计方法规范》第7节）
- [ ] **跨模块关联矩阵已检查**：涉及多模块交叉的功能已绘制资源状态×关联功能矩阵，标记"需验证"的交叉点均有对应用例

# 需求文档生成规范

> 版本：v1.0 | 生效日期：2026-07-17 | 所属：AI模型平台测试规范体系
>
> **目的**：打通需求文档生成与测试解析之间的格式链路，确保上游产出格式与下游解析期望完全对齐。

---

## 1. 规范适用范围

本规范适用于以下场景：
- 使用 `generate-prd` 技能从知识库生成需求文档
- 手动编写 Sprint PRD 文档
- 需求文档格式审查与格式校验

---

## 2. 需求编号规范

### 2.1 功能模块编号

**格式**：`F{N}`，其中 N 为正整数

**示例**：`F1`、`F2`、`F16`

**命名规则**：
- 功能编号按 Sprint 内功能顺序连续编排
- 同一 Sprint 内功能编号唯一
- 跳号时必须在文档开头说明原因和去向

### 2.2 产品规则编号

**格式**：`PR-{模块号}.{序号}`

**示例**：`PR-4.1`、`PR-8.3`

### 2.3 验收标准编号

**格式**：`AC-{模块号}.{序号}`

**示例**：`AC-1.1`、`AC-16.10`

**编号规则**：
- 编号连续，不重复，不留空号
- 按功能逻辑分组时，编号仍保持全局连续

### 2.4 待确认事项编号

**格式**：`T-{序号}`

**示例**：`T-01`、`T-18`

### 2.5 产品目标编号

**格式**：`G{序号}`

**示例**：`G1`、`G6`

### 2.6 用户故事编号

**格式**：`US-{序号}`

**示例**：`US-2`、`US-10`

## 3. 需求文档标准结构

所有 Sprint 需求文档（自 Sprint 4 v5.0 起）必须遵循统一 8 子节模板，每个 F 模块（`### FN. 模块名`）包含：

| 子节 | 内容要求 | 格式规范 |
|------|---------|---------|
| 8.N.1 需求说明 | 功能目标+触发条件+前置依赖+涉及角色+映射产品目标(G1~GN) | 段落描述 |
| 8.N.2 产品规则 | 原子化规则列举，每条独立子弹点 | 格式 `[PR-N.X] 描述` |
| 8.N.3 边界条件与异常场景 | 四分区：输入边界/并发竞态/降级容灾/存量兼容 | 缺失项标 `[待确认:T-XX]` |
| 8.N.4 字段定义 | 三层字段表：页面展示→API→数据库/索引 | 均含「是否本期新增」列 |
| 8.N.5 页面交互 | 操作入口+交互流程+二次确认规则+提示/错误文案 | 入口精确到菜单，文案精确到字 |
| 8.N.6 状态流转 | Mermaid stateDiagram-v2 + 转移条件表 | 无生命周期则声明 |
| 8.N.7 验收标准 | AC编号 + Given-When-Then 格式 | 一条一个断言 |
| 8.N.8 补充约束 | 安全口径/审计/i18n/部署开关等 | 格式 `[约束编号] 类别 \| 描述 \| 影响范围` |

### 3.1 验收标准格式

**当前格式（Sprint4及之前）**：

```markdown
#### 8.X.7 验收标准

- [ ] 平台把公共接入点 ep-public 授权给租户 A 后，租户 A 普通用户立即可见、可绑定、可调用 ep-public（授权即生效，无租户侧开启环节）。
- [ ] 平台把模型 M 授权给租户 A 后，租户 A 普通用户可在模型广场看到模型 M，可创建模型级 API Key，可把模型 M 作为业务接入点候选，可直调 `model=M`。
```

**推荐格式（Sprint5起采用）**：

```markdown
#### 8.X.7 验收标准

### 一、分组标题（AC-X.1 ~ AC-X.N）

- [ ] **AC-X.1** Given 前置条件；When 操作动作；Then 预期结果。
- [ ] **AC-X.2** Given 前置条件；When 操作动作；Then 预期结果。

### 二、分组标题（AC-X.N+1 ~ AC-X.M）

- [ ] **AC-X.N+1** Given 前置条件；When 操作动作；Then 预期结果。
```

**格式规则**：
1. Sprint4及之前：使用 `- [ ]` 复选框格式，内容描述完整清晰
2. Sprint5起：使用 `[ ] **AC-X.N** Given...When...Then...` 格式
3. 编号连续，不重复，不留空号（仅推荐格式）
4. 删除纯后端数据的验收标准（如审计日志字段验证），仅保留测试工程师需关注的验证点
5. 去重：合并内容重复或高度相似的验收标准
6. 仅保留与当前功能直接相关的验收标准，跨功能关联内容放入对应功能模块

## 4. 需求文档分拆结构规范

### 4.1 目录结构

从知识库解析需求到本地项目文档时，采用「一个迭代一个文件夹」的结构：

```
需求文档/
├── Sprint1_需求文档.docx                    ← Sprint1~3 使用 Word 格式存档
├── Sprint2_需求文档.docx
├── Sprint3_需求文档.docx
├── Sprint4_需求文档_分拆/                    ← Sprint4 起使用 Markdown 分拆结构
│   ├── README.md                            ← 索引文件，说明分拆结构与原文件信息
│   ├── 00_文档头与通用章节.md                ← §1~§7 通用章节
│   ├── 01_F1_平台资源池与租户授权范围.md     ← F1 功能模块
│   ├── 02_F2_接入点渠道锁定与路由意图.md     ← F2 功能模块
│   ├── 03_F4_转发侧租户账号与API_Key同步.md ← F4 功能模块
│   ├── ...                                  ← 其他功能模块文件
│   └── 14_非功能需求与收尾.md               ← §9~§13 非功能需求与收尾
├── Sprint5_需求文档_分拆/                    ← 后续 Sprint 按相同结构
│   ├── README.md
│   ├── 00_文档头与通用章节.md
│   ├── 01_F1_xxx.md
│   └── ...
```

### 4.2 文件命名规范

| 文件类型 | 命名格式 | 示例 |
|---------|---------|------|
| 索引文件 | `README.md` | `README.md` |
| 文档头与通用章节 | `00_文档头与通用章节.md` | `00_文档头与通用章节.md` |
| 功能模块文件 | `{两位序号}_{功能编号}_{功能名称}.md` | `01_F1_平台资源池与租户授权范围.md` |
| 非功能需求与收尾 | `{两位序号}_非功能需求与收尾.md` | `14_非功能需求与收尾.md` |

**命名规则**：
- 序号从 `00` 开始，两位数字，连续递增
- 功能编号使用 `F{N}` 格式
- 功能名称使用中文，下划线分隔
- 文件名为 ASCII 字符，避免特殊符号

### 4.3 README.md 内容要求

README.md 作为索引文件，必须包含以下内容：

```markdown
# Sprint N 需求文档 — {主题}（分拆索引）

## 原文件信息

- **原文件位置**：{知识库中PRD的相对路径}
- **版本**：vX.Y（基于测试工程师视角全量重构）
- **生成日期**：YYYY-MM-DD
- **需求权威来源**：{知识库中PRD的完整路径}

## 分拆结构

| 文件 | 内容范围 | 说明 |
|------|----------|------|
| `README.md` | 索引文件 | 说明分拆结构、各文件用途、原文件位置 |
| `00_文档头与通用章节.md` | §1~§7 | 文档头元信息、一句话说明、背景、产品目标、非目标、用户与角色、核心用户场景、范围总览 |
| `01_F1_xxx.md` | F1（§8.1.1~8.1.8） | {功能描述} |
| ... | ... | ... |
| `NN_非功能需求与收尾.md` | §9~§13 | 非功能需求、总体验收清单、待确认事项、建议拆分、版本记录 |

## 说明

- 各文件保留完整的8子节结构：需求说明/产品规则/边界条件/字段定义/页面交互/状态流转/验收标准/补充约束
- 分拆后各文件独立可阅读，同时保留交叉引用关系
```

### 4.4 文档头与通用章节（00_文档头与通用章节.md）

包含以下章节：
- §1 一句话说明
- §2 背景
- §3 产品目标（G1~GN）
- §4 非目标
- §4a 移出本期内容清单（如有）
- §5 用户与角色（含用户故事 US-*）
- §6 核心用户场景
- §7 范围总览（含菜单变动）

文件末尾必须包含后续章节的交叉引用：
```markdown
**后续章节见分拆文件：**
- [01_F1_平台资源池与租户授权范围.md](01_F1_平台资源池与租户授权范围.md)
- [02_F2_接入点渠道锁定与路由意图.md](02_F2_接入点渠道锁定与路由意图.md)
- ...
- [NN_非功能需求与收尾.md](NN_非功能需求与收尾.md)
```

### 4.5 功能模块文件（01_F1_xxx.md ~ N-1_FN_xxx.md）

每个功能模块文件必须包含完整的8子节结构：
- 8.N.1 需求说明（功能目标+触发条件+前置依赖+涉及角色+映射产品目标）
- 8.N.2 产品规则（原子化规则列举）
- 8.N.3 边界条件与异常场景（四分区）
- 8.N.4 字段定义（三层字段表）
- 8.N.5 页面交互（操作入口+交互流程+二次确认规则+提示/错误文案）
- 8.N.6 状态流转（Mermaid stateDiagram-v2 + 转移条件表）
- 8.N.7 验收标准（分组标题 + AC编号 + Given-When-Then格式）
- 8.N.8 补充约束（安全口径/审计/i18n/部署开关等）

文件末尾必须包含上下章节的交叉引用：
```markdown
**上一章节**：[00_文档头与通用章节.md](00_文档头与通用章节.md)
**下一章节**：[02_F2_接入点渠道锁定与路由意图.md](02_F2_接入点渠道锁定与路由意图.md)
```

### 4.6 非功能需求与收尾（NN_非功能需求与收尾.md）

包含以下章节：
- §9 非功能需求
- §10 总体验收清单
- §11 待确认事项（T-*）
- §12 建议拆分（如有）
- §13 版本记录

## 5. 格式质量红线

1. **零乱码**：禁止字符重复、HTML实体、转义复选框残留
2. **编号连续**：禁止跳号无说明、禁止字母后缀（8.2.3a 等）
3. **缺号说明**：功能编号跳号时必须在 §8 开头引用块说明原因和去向
4. **规则原子化**：每条独立可测试
5. **验收标准 Given 必完备**：禁止省略前置条件
6. **字段定义三层覆盖**：页面展示→API→数据库/索引
7. **边界条件四分区至少各一条**：输入边界/并发竞态/降级容灾/存量兼容

## 6. 与测试解析规范的对齐关系

本规范定义的格式必须与《需求解析规则规范》第7节中的需求ID提取正则完全匹配：

| 本规范定义的格式 | 需求解析规则规范中的正则 | 匹配示例 |
|-----------------|-----------------------|---------|
| F{N} | `F[-_]?\d+(\.\d+)*` | F1, F-3.1.2 |
| AC-X.Y | 由解析规则自动识别 | AC-1.1, AC-16.10 |
| PR-X.Y | 由解析规则自动识别 | PR-4.1, PR-8.3 |
| G{N} | 由解析规则自动识别 | G1, G6 |
| US-{N} | `US[-_]?\d{3,}` | US_042, US-001 |

**一致性要求**：
1. 需求文档生成时必须使用本规范定义的编号格式
2. 测试解析时必须能正确识别上述所有格式
3. 格式变更必须同步更新本规范和《需求解析规则规范》

## 7. 与 generate-prd 技能的集成

`generate-prd` 技能在生成需求文档时必须：

1. **读取本规范**：在生成前读取《需求文档生成规范》
2. **使用标准格式**：使用 `F{N}` 格式进行功能编号，使用 `AC-X.Y` 格式进行验收标准编号
3. **生成标准结构**：生成的PRD必须包含8.N.1~8.N.8完整子节
4. **输出格式检查**：生成后对照本规范进行格式自检

---

## 8. 验收标准检查清单

生成需求文档后，执行以下检查：

- [ ] 功能编号使用 `F{N}` 格式
- [ ] 验收标准编号使用 `AC-X.Y` 格式，且连续无跳号
- [ ] 产品规则编号使用 `PR-X.Y` 格式
- [ ] 每个功能模块包含完整的8.N.1~8.N.8子节
- [ ] 验收标准使用 `Given...When...Then...` 格式
- [ ] 边界条件覆盖四分区（输入边界/并发竞态/降级容灾/存量兼容）
- [ ] 字段定义包含三层（页面展示→API→数据库/索引）
- [ ] 无乱码、无跳号无说明、无字母后缀编号

# 需求追溯矩阵规范

> 版本：v1.0 | 生效日期：2026-06-11 | 所属：AI模型平台测试规范体系

---

## 1. 追溯矩阵概述

### 1.1 目的

1. **双向追溯**：从需求找用例，从用例找需求
2. **覆盖验证**：确保每个需求都有测试用例覆盖
3. **变更影响**：需求变更时快速定位受影响的用例
4. **遗漏检测**：发现未覆盖的需求和孤儿用例

---

## 2. 追溯矩阵结构

### 2.1 正向追溯（需求 → 用例）

| 需求ID | 需求名称 | 验收条件数 | 用例ID列表 | 用例数量 | 覆盖状态 |
|--------|---------|-----------|-----------|---------|---------|
| REQ_001 | 用户登录功能 | 3 | 用例列表（逗号分隔） | 5 | ✅ 已覆盖 |
| REQ_002 | 密码找回功能 | 2 | 用例列表 | 2 | ✅ 已覆盖 |
| REQ_003 | 第三方登录 | 2 | - | 0 | ❌ 未覆盖 |

### 2.2 反向追溯（用例 → 需求）

在测试用例Excel中增加「关联需求」列：

| 用例标题 | 关联需求ID | 需求名称 |
|---------|-----------|---------|
| 【# REQ_001_LOGIN_账号密码登录】正向：正常登录 | REQ_001 | 用户登录功能 |

| 无关联ID的用例 | - | ⚠️ 孤儿用例 |

## 3. 需求ID提取规则

> 详细规则请参阅《需求解析规则规范》第7节，此处仅保留自动生成规则和层级定义。

### 3.1 自动生成规则

如果需求文档无明确编号，按以下规则自动生成：

```
格式：MOD_{模块缩写}_{序号}

示例：
- 登录模块第1个需求：MOD_LOGIN_001
- 订单模块第3个需求：MOD_ORDER_003
- 模型管理第1个需求：MOD_MODEL_001

AI模型平台核心模块缩写：
- 模型管理 → MODEL
- 模型部署 → DEPLOY
- 模型数据 → DATA
- 模型观测 → OBSERVE
- API管理 → API
- API Key管理 → APIKEY
- 安全围栏 → SEC
- 租户管理 → TENANT
- 接入点 → ACCESS
- 模型市场 → MARKET
- 模型实验室 → LAB
- OM点数包 → OM
```

### 3.2 需求层级

```
模块 (Module)
  └── 功能 (Feature)
        └── 需求点 (Requirement)
              └── 验收条件 (Acceptance Criteria)
```

追溯到**需求点**级别，验收条件体现在用例步骤中。

## 4. 覆盖率计算

### 4.1 需求覆盖率

```
需求覆盖率 = 已覆盖需求数 / 总需求数 × 100%
```

**计算规则**：
- 需求至少有1条用例关联，即视为"已覆盖"
- 建议覆盖率目标：≥95%

### 4.2 覆盖深度

```
覆盖深度 = 总用例数 / 总需求数
```

**参考标准**：

| 模块风险等级 | 建议覆盖深度 |
|-------------|-------------|
| 核心/高风险 | ≥5 用例/需求 |
| 一般功能 | 3-5 用例/需求 |
| 边缘功能 | 1-3 用例/需求 |

### 4.3 验收条件覆盖率

```
验收覆盖率 = 已覆盖验收条件数 / 总验收条件数 × 100%
```

每个验收条件应至少有1条正向用例验证。

## 5. 遗漏检测

### 5.1 未覆盖需求检测

**检测逻辑**：
1. 遍历需求列表
2. 对每个需求ID，查找关联的用例
3. 若用例数为0，标记为「未覆盖」

**输出格式**：

```markdown
## 未覆盖需求清单

| 需求ID | 需求名称 | 风险等级 | 建议操作 |
|--------|---------|---------|---------|
| REQ_003 | 第三方登录 | 高 | 需补充用例 |
| REQ_015 | 消息推送 | 中 | 需补充用例 |

未覆盖需求数：2
总需求数：20
覆盖缺口：10%
```

### 5.2 孤儿用例检测

**定义**：无关联需求ID的用例

**检测逻辑**：
1. 遍历用例列表
2. 检查「关联需求ID」列
3. 若为空或无效ID，标记为「孤儿用例」

**处理建议**：
- 尝试匹配需求（可能是遗漏标记）
- 若确实无对应需求，考虑删除或归档

## 6. 追溯矩阵生成流程

### Phase 1: 解析需求文档

```
输入：需求文档（PDF/Word/Markdown）
输出：需求列表
```

1. 提取文本
2. 按照「需求ID提取规则」识别需求
3. 提取每个需求的：需求ID、需求名称/标题、验收条件、所属模块

### Phase 2: 生成用例并关联

```
输入：需求列表、测试设计方法
输出：带关联的测试用例
```

1. 对每个需求生成用例
2. 每个用例记录来源需求ID
3. 存入Excel「关联需求ID」列

### Phase 3: 生成追溯矩阵

```
输入：需求列表、用例列表
输出：追溯矩阵表
```

1. 汇总需求-用例对应关系
2. 计算覆盖率指标
3. 标记遗漏项

## 8. 覆盖检查清单

生成追溯矩阵后，执行以下检查：

- [ ] 每个需求至少有1条正向用例
- [ ] 每个需求的验收条件都有对应用例
- [ ] 核心需求（P0）覆盖深度 ≥5
- [ ] 无孤儿用例
- [ ] 需求覆盖率 ≥95%
- [ ] 高风险模块有错误推测用例补充

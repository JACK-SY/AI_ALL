## 1. 一句话说明

Sprint 4 要把模型平台从"可以调用模型"升级为"可授权、可治理、可审计、可运营的模型调用平台"。本期重点不是新增模型训练能力，而是补齐模型被纳管后如何被租户发现、授权、调用、记录、统计和清理的闭环。

***

## 2. 背景

当前平台已经具备模型广场和基础 API Key 调用能力，但治理能力不足：

| 问题           | 现状影响                                                  |
| ------------ | ----------------------------------------------------- |
| 模型默认可见边界不清   | 模型上架后缺少租户级授权控制，平台管理员无法精确控制哪些租户可见、可绑定、可调用              |
| 接入点职责混在一起    | 平台维护的公共入口、租户自建的业务入口和租户资源开关没有清晰边界，路由配置、资源启用状态和维护责任容易混淆 |
| API Key 治理不足 | 大模型、小模型凭证混杂；权限粒度不足；资源关闭、人员离职后缺少闭环处置                   |
| 调用日志事实源薄弱    | 调用日志直接依赖查询存储，缺少可重放、多消费者的事实流，影响排障、审计、用量和计费核对           |
| 用量和商业资格未打通   | 不同角色看到的用量范围不清；OM 点数包、试用和白名单未形成统一调用资格判断                |
| 用户级成本限额缺失    | AI Coding 等高消耗场景缺少按用户月度 Token 控制能力，异常个人使用可能放大全租户或平台成本 |

***

## 3. 产品目标

| 编号 | 目标            | 成功标准                                                                       |
| -- | ------------- | -------------------------------------------------------------------------- |
| G1 | 租户资源启用边界清晰    | 平台给出租户可配置资源池；租户管理员可以分别开启或关闭模型和公共接入点；关闭资源不可见、不可绑定、不可调用                      |
| G2 | 接入点边界清晰       | 公共接入点由平台维护，业务接入点由租户维护；两类接入点的权限和路由规则不同                                      |
| G3 | API Key 可治理   | 大模型接入凭证独立管理；支持全部资源权限和部分资源权限；资源关闭后可追踪失效绑定                                   |
| G4 | 离职资源可清理       | 平台能识别离职人员创建的 API Key，并按租户策略禁用或删除                                           |
| G5 | 调用事实可追溯       | 调用明细事件进入 Kafka，ES 仅作为查询索引；调用日志、用量、告警和计费核对共享同一事实来源                          |
| G6 | 用量、调用资格和限额可运营 | 普通用户、租户管理员、平台管理员看到不同范围的用量；租户调用前受 OM 点数包、试用或白名单资格控制；平台可对用户月度 Token 用量进行限额治理 |

## 4. 非目标

- 模型训练、训练任务、训练数据集、训练产物、微调流程
- 用户级模型开通流程（申请接入、审批、允许名单、按用户限制资源使用、撤销开通）；"按用户限制资源使用"指资源开通和允许名单，不包含本期用户级用量限额
- 租户自助购买 OM 点数包页面、订单、发票、云市场售卖配置
- 完整租户自助限额配置页面、限额阈值通知和短时间窗口 RPM / TPM 调用限流；本期用量限额先支持后台配置和月度 Token 成本治理

- AI 安全护栏独立一级菜单和复杂策略编排
- 资源关闭失效绑定的手动重新生效、重新选择资源或批量清理入口
- **资源关闭失效绑定的展示与自动恢复逻辑（本期不做，推迟至后续迭代）**
- **F3 大模型接入凭证重设计（推迟至后续迭代）**
- **F6 调用资格逻辑（推迟至后续迭代）**
- **F6 长解方案——模型成本归属、入口无关成本限额、路由后限额校验（推迟至后续迭代）**

## 5. 用户与角色

| 角色           | 主要诉求                                          |
| ------------ | --------------------------------------------- |
| 普通用户 / 开发者   | 发现本租户可用模型和接入点，创建自己的大模型接入凭证，查看自己的调用日志和用量       |
| 租户管理员        | 管理本租户业务接入点、API Key、用量、离职资源和治理联系人              |
| 平台管理员        | 纳管模型供应商和模型，维护公共接入点和平台资源池默认启用策略，查看全平台调用事实和治理状态 |
| 网关 / 转发侧系统   | 执行 API Key 校验、授权校验、路由、转发、日志事件上报和缓存同步          |
| OM 系统        | 提供租户调用资格来源，消费调用明细事件完成点数扣减或对账                  |
| IAM / 组织身份系统 | 提供离职人员名单，模型平台不自行判定离职状态                        |

### 用户故事

| 编号   | 故事                                                            |
| ---- | ------------------------------------------------------------- |
| US-1 | 作为租户管理员，希望在平台给到的资源池内开启或关闭本租户可用的模型和公共接入点，以便控制普通用户能看到、绑定和调用哪些资源 |
| US-2 | 作为租户管理员，希望平台自动识别离职员工创建的 API Key 并提示处理，以便消除安全和费用风险             |
| US-3 | 作为租户内开发者，希望通过公共接入点调用模型，以便不需要关心背后模型的具体授权和供应商细节                 |
| US-4 | 作为平台管理员，希望查看每次调用的详细日志，以便支撑排障、审计和计费核对                          |
| US-5 | 作为平台管理员，希望在模型或公共接入点加入租户资源池时配置默认启用策略，以便兼顾平台运营效率和租户管理员治理权       |
| US-6 | 作为平台管理员，希望限制高消耗租户中具体用户的月度 Token 用量，以便快速控制异常模型调用成本             |

## 6. 核心用户场景

### 场景1：平台管理员纳管模型并分配租户

1. 平台管理员在「模型管理 → 模型库」确认纳管模型信息
2. 选择进入哪些租户的平台资源池，设置默认启用/关闭
3. 可选勾选生成公共接入点，配置路由策略和 AI 安全护栏推荐配置
4. 提交→资源进入对应租户的平台资源池→是否立刻对普通用户可见取决于默认启用策略和租户管理员后续配置

### 场景2：创建业务接入点

> **已澄清**：S4 中普通用户和租户管理员均可创建/编辑业务接入点。

1. 用户在「接入点」新建业务接入点
2. 选择本租户已启用模型作为候选（只展示 modelId + providerName）
3. 配置完成后调用方可通过 ep-\* 目标调用

### 场景3：开发者创建凭证并调用

1. 开发者在「基础设置 → 大模型接入凭证」创建 API Key
2. 选择权限范围（全部资源 or 部分资源），绑定模型和/或接入点
3. 使用 API Key 发起调用→前置校验通过→路由→转发→返回响应

### 场景4：租户管理员排查异常调用

1. 在「调用日志」筛选异常状态
2. 查看详情：实际命中模型、供应商来源、Token、点数、耗时和安全护栏结果
3. 必要时在「用量统计」按维度钻取分析

### 场景5：租户管理员处理离职资源

1. 平台定时同步 IAM 离职人员→离职 API Key 自动标记
2. 租户管理员在「离职资源清理」查看待处理列表
3. 确认删除→审计日志记录

## 7. 范围总览

```
模型纳管 → 平台资源池 → 租户资源管理 → 模型发现/接入点 → API Key 鉴权 → 网关调用 → 调用日志 → 用量统计 → 控制台展示 → 公开 API 文档
```

## 8. 功能需求

### F1. 平台资源池与租户资源管理

**优先级**：P0

平台需要先给租户形成可配置的资源池，再由租户管理员决定本租户开启或关闭哪些模型和公共接入点。普通用户只看到已启用资源。

#### 资源类型

- **模型**：来源为平台预先分配给租户的租户可选模型池；租户管理员可开启或关闭模型
- **公共接入点**：来源为平台创建公共接入点并加入租户资源池；租户管理员可开启或关闭公共接入点
- 模型和公共接入点是**两类独立资源**。关闭公共接入点不联动关闭其背后候选模型；关闭模型不影响已开启公共接入点的内部路由

#### 菜单与配置位置

| 操作       | 位置               | 角色                |
| -------- | ---------------- | ----------------- |
| 模型库维护    | 模型管理 → 模型库       | 平台管理员 / Studio工具组 |
| 平台资源池维护  | 模型或公共接入点上架/分配租户时 | 平台管理员             |
| 默认启用策略   | 资源上架、分配租户或加入资源池时 | 平台管理员             |
| 租户资源启用配置 | **资源管理**（新增一级菜单） | 租户管理员             |
| 接入点维护    | **接入点**          | 平台管理员、租户管理员       |

> 默认启用策略只影响资源**首次**进入租户资源池。租户管理员手动调整后，后续平台更新或重新分配同一资源**不得覆盖**租户管理员配置。

#### 资源管理页面

建议两个页签：

**模型页签**：

| 字段           | 说明          |
| ------------ | ----------- |
| modelId      | 模型标识        |
| providerName | 供应商名称       |
| 能力标签         | 模型能力分类      |
| 来源公共接入点      | 该模型首次可见的接入点 |
| 启用状态         | 已启用/已关闭     |
| 更新时间         | 最近状态变更时间    |
| 操作人          | 最近操作人       |
| 操作           | 开启、关闭       |

**公共接入点页签**：

| 字段     | 说明                          |
| ------ | --------------------------- |
| 接入点 ID | 唯一标识                        |
| 名称     | 接入点名称                       |
| 路由意图   | 负载均衡/健康降级/复杂度路由/顺序/权重       |
| 默认模型摘要 | 代表模型 modelId + providerName |
| 启用状态   | 已启用/已关闭                     |
| 更新时间   | 最近状态变更时间                    |
| 操作人    | 最近操作人                       |
| 操作     | 开启、关闭                       |

**租户管理员不能**：从全平台模型库自行加入模型、编辑公共接入点信息、修改公共接入点候选模型、替普通用户创建 API Key

#### 租户侧展示结果

- **模型广场**：普通用户只展示本租户已启用且可调用的模型
- **模型详情**：展示能力、供应商来源摘要、可用状态和模型级调用方式；关闭模型普通用户不可见
- **接入点列表**：普通用户只展示已启用公共接入点和自己可见的业务接入点
- **公共接入点详情**：只读展示调用方式、状态、AI 安全护栏状态、调用日志入口、路由意图和当前默认模型摘要
- **API Key 绑定选择器**：只展示已启用模型和已启用接入点
- **API Key 详情**：展示有效绑定和因资源关闭产生的失效绑定（只读）

#### 产品规则

1. 平台资源池限制租户管理员可配置范围；租户管理员只能在平台已分配给本租户的资源池内操作，不能超出资源池范围自行开启全平台模型或接入点（即租户隔离 + 资源池范围双重约束）
2. 租户可选模型池由平台预先分配；首版通过 F9 发布流程完成分配，平台管理员在发布纳管模型时选择进入哪些租户的平台资源池并设置默认启用状态
3. 模型和公共接入点启用状态独立配置
4. 关闭公共接入点只影响该公共接入点在本租户下的可见、绑定和调用，不影响背后候选模型
5. 关闭模型只影响模型广场可见、模型级 API Key 绑定、模型直调和业务接入点新增候选，不影响已开启公共接入点的调用
6. 业务接入点创建或编辑时只能选择本租户已启用模型
7. 租户管理员关闭某个模型后，业务接入点中已有该模型候选进入不可用或异常状态；系统**不自动替换**成其他模型
8. 关闭模型或公共接入点后，已有 API Key 绑定保留但进入**资源关闭失效绑定**状态（注：**资源关闭失效绑定本期不做**，本期关闭资源后绑定逻辑不变，失效绑定的展示与恢复推迟至后续迭代）
9. 重新开启同一资源后，历史失效绑定可自动恢复有效；不同资源不得自动复用旧绑定
10. 资源关闭、开启、默认启用策略和 API Key 失效绑定变化都需记录审计日志

#### 验收标准

1. 平台把公共接入点 ep-public 加入租户 A 资源池并设置默认启用后，租户 A 普通用户可见、可绑定、可调用 ep-public
2. 平台把模型 M 加入租户 A 可选模型池并设置默认关闭后，租户 A 普通用户默认不可见、不可绑定、不可直接调用模型 M（默认关闭是在分配资源的**同时**设置的，而非先加入再单独关闭）
3. 租户管理员开启模型 M 后，普通用户可在模型广场看到模型 M，可创建模型级 API Key，可作为业务接入点候选（注：租户管理员的操作是"启用资源开关"而非"发布模型"，模型广场的展示是被动的——跟随开关状态变化，不是主动发布动作）
4. 租户管理员关闭模型 M 后，模型广场不展示模型 M，API Key 不能新增绑定模型 M，直接调用 `model=M` 被拒绝
5. 租户管理员关闭模型 M 后，已开启公共接入点 ep-public 仍可按平台路由内部命中模型 M
6. 租户管理员关闭公共接入点 ep-public 后，不联动关闭模型 M
7. 租户管理员关闭公共接入点 ep-public 后，API Key 不能新增绑定 ep-public，已绑定 ep-public 的 API Key 调用被拒绝，并展示资源关闭失效绑定  
8. 租户管理员重新开启同一资源后，历史资源关闭失效绑定自动恢复有效
9. 平台重新分配或更新同一资源时，不覆盖租户管理员已经手动调整过的启用状态
10. 业务接入点创建或编辑时只能选择本租户已启用模型
11. 租户管理员关闭业务接入点候选模型后，系统不自动替换成其他模型

### F2. 接入点归属类型、路由意图与路由策略

**优先级**：P0

区分公共接入点和业务接入点两类归属类型。首版仅在运营租户维护的公共接入点中配置路由意图（负载均衡、健康降级、复杂度路由）；业务接入点先不开放路由意图配置。

#### 接入点归属类型

- **公共接入点**：平台管理员维护，面向多租户使用的调用入口，租户只能使用和查看摘要，不能编辑路由
- **业务接入点**：租户管理员维护，面向单租户业务系统或场景入口，归属当前租户，只能选择本租户已启用模型

#### 路由意图

| 意图    | 说明                                           | 首版策略                                                   |
| ----- | -------------------------------------------- | ------------------------------------------------------ |
| 负载均衡  | 同一模型能力由多个供应商提供，候选应为平台管理员人工确认的同一模型能力          | 随机或轮询等价分流，默认随机，不配置权重                                   |
| 健康降级  | 主模型不可用或出现可重试失败时切换到备用或降级模型，允许不同模型能力           | 必须明确主模型、降级模型和降级顺序                                      |
| 复杂度路由 | 调用方通过 `X-Task-Complexity` Header 显式传入任务复杂度标记 | 固定 `simple`、`normal`、`complex` 三档，每档选择一个模型，默认 `normal` |

> - 负载均衡意图下，系统不只靠 `modelId` 字符串自动判断"同一模型能力"，需平台管理员人工确认。**举例**：`deepseek-r1`（供应商A）与 `deepseek-r1`（供应商B）因 modelId 相同算同一模型能力；但 `deepseek-r1` 与 `qwen3.5` 即使能力等价也不算——负载均衡要求候选的 modelId 必须相同，仅 providerId 不同，由平台管理员确认它们确实提供等价服务后纳入同一负载均衡组
> - 复杂度路由缺少 Header 时走默认档位 `normal`；非法标记按请求参数错误拒绝，不回退默认档位
> - 业务接入点首版不开放负载均衡、健康降级和复杂度路由意图配置

#### 租户侧展示与操作

- 租户侧使用一个列表展示公共接入点和业务接入点，通过类型字段和筛选区分
- 公共接入点租户侧只读：
  - 详情展示调用方式、状态、AI 安全护栏状态、调用日志入口、路由意图和当前默认模型摘要
  - 复杂度路由时展示允许的复杂度标记、默认档位和每档用途说明
  - 可能发生健康降级时展示"可能降级"提示
  - 默认模型摘要展示 `modelId` + `providerName`
- 公共接入点详情租户不可见：完整候选列表、顺序优先级、负载均衡候选列表、轮询游标、候选数量、供应商配置明细
- 业务接入点路由候选选择器只展示 modelId + providerName，不展示供应商账号、服务地址、成本、内部转发配置和可用状态
- 业务接入点已绑定候选被关闭或不可用后该候选进入不可用或失效状态；系统不自动替换；租户管理员需手动编辑恢复；所有候选失效时调用失败

#### 路由规则

1. 接入点路由候选绑定键按 `providerId + modelId` 理解
2. 公共接入点配置必须包含归属类型、路由意图和路由策略；业务接入点不包含
3. 负载均衡意图下路由策略支持随机和轮询；轮询需转发侧具备可维护游标状态缓存能力
4. 路由策略运行条件不满足时如无多个模型，不允许配置轮询或必须给出明确提示
5. 健康降级意图按主模型→降级模型→降级顺序依次尝试
6. 复杂度路由解析 `X-Task-Complexity` 得到档位，命中最该档位唯一模型；首版不在档位内继续做负载均衡
7. **可重试失败**不自动切换候选模型，而是返回错误码让调用方决定是否重试：上游超时、连接失败、上游 5xx、供应商限流 429、模型临时不可用
8. **不可重试失败**不触发切换：权限不足、请求参数错误、余额不足、AI 安全护栏拦截、内容合规拦截、模型业务拒答
9. 实际命中模型、路由意图、是否降级、复杂度请求/解析/默认值必须写入调用明细事件

#### 验收标准

1. 平台管理员可创建、编辑、禁用公共接入点
2. 租户管理员不可编辑公共接入点路由和候选模型配置
3. 租户侧一个列表展示公共接入点和业务接入点，可按类型筛选
4. 公共接入点必须配置路由意图
5. 业务接入点首版不开放路由意图配置
6. 公共接入点详情租户侧只读，不展示完整候选列表等
7. 负载均衡候选由平台管理员人工确认为同一模型能力
8. 负载均衡首版支持随机和轮询，默认随机
9. 轮询条件不满足时系统不允许配置或给出提示
10. 健康降级下可配置主模型和降级模型；降级时调用日志记录
11. 复杂度路由租户侧可查看允许标记、默认档位和用途
12. 支持 `X-Task-Complexity` 三档；缺少走默认；非法按参数错误拒绝
13. 复杂度路由日志记录传入标记、最终档位和是否使用默认
14. 多候选路由时租户侧只展示默认模型摘要；实际命中模型在调用日志展示
15. 租户管理员可创建、编辑本租户业务接入点
16. 业务接入点只能选择本租户已启用模型
17. 候选选择器只展示 modelId 和 providerName
18. 候选模型关闭或不可用后系统不自动替换
19. 所有候选失效时调用失败，提示需编辑路由候选
20. 健康降级主模型超时后切换到降级模型
21. 权限不足等不可重试失败不触发模型切换

### F3. 大模型接入凭证重设计

**优先级**：P0

> **⚠️ 已澄清：F3 大模型接入凭证重设计本期不做，推迟至后续迭代。** 以下内容保留作为需求基线参考，不进入 S4 开发与测试范围。

将大模型调用凭证从现有 API Key 管理中独立出来。同一把 API Key 可同时绑定模型和接入点，支持全部资源权限（动态扩展）和部分资源权限（显式白名单）。

#### 产品规则

1. 大模型接入凭证本质是 API Key，归属租户和真实创建人
2. 同一把 API Key 可同时绑定模型和接入点
3. 模型绑定按具体纳管模型选择，后台按 `providerId + modelId` 保存和校验，页面展示 `modelId` + `providerName`
4. 绑定的模型或公共接入点被租户管理员关闭后，该资源绑定进入**资源关闭失效绑定**；绑定记录保留用于排查，但对应目标调用立即失败
5. 模型绑定失效时失效对象展示为模型（注：**资源关闭失效绑定本期不做**，此展示逻辑推迟至后续迭代）
6. API Key 绑定接入点时，接入点背后候选模型不可用、下架、删除或恢复，**不改变**该接入点绑定状态，也不产生资源关闭失效绑定
7. 同一 `providerId + modelId` 后续重新启用并满足条件时，资源关闭失效绑定可自动恢复有效，需同步 Go token 并清理 Redis
8. 若恢复的是不同 `providerId` 或不同 `modelId`，旧失效绑定**不得自动恢复**
9. **全部资源权限**：对归属租户当前和未来已启用模型、接入点动态生效；新增启用资源后不需要逐把更新；不绕过资源启用状态、OM 调用资格、余额和 AI 安全护栏
10. **部分资源权限**：只对显式选择的模型和接入点生效（白名单）；新增启用资源不自动进入
11. 调用目标以 `ep-*` 开头时按接入点校验，否则按模型校验
12. 调用接入点时不要求 API Key 同时启用或绑定背后候选模型
13. API Key 禁用、启用、删除、权限变化、失效绑定产生后都必须同步 Go 转发侧并清理 Redis
14. 资源关闭失效绑定只影响对应目标，不改变 API Key 整体状态
15. v1 只读展示失效绑定，不提供手动重新生效或清理入口

#### 列表字段

凭证名称、所属租户、创建人、状态（启用/禁用/已删除/同步中/同步失败）、权限范围、失效绑定数量、供应商来源、最近调用时间、本月点数消耗

#### 详情字段

API Key 基本信息、有效权限范围、失效绑定列表（资源类型/资源名称/失效原因/产生时间/转发侧同步状态）、最近调用记录、审计记录

#### 验收标准

1. 导航上独立展示大模型接入凭证和小模型服务凭证
2. 大模型接入凭证可同时绑定模型和接入点
3. 部分资源权限绑定模型时必须绑定到具体纳管模型
4. 全部资源权限的 API Key 在租户新增启用资源后自动可调用
5. 部分资源权限的 API Key 在租户新增启用资源后不可自动调用
6. 调用目标 `ep-*` 按接入点校验
7. 调用接入点时不要求 API Key 启用或绑定背后候选模型
8. 绑定的资源被关闭后进入失效绑定，调用立即失败
9. 详情展示失效绑定原因、时间和同步状态
10. 接入点背后候选不可用不改变接入点绑定状态
11. 失效绑定只影响对应目标，其他绑定继续可用
12. 同一 `providerId + modelId` 重新启用后失效绑定自动恢复
13. 不同 `providerId` 或 `modelId` 恢复时旧失效绑定不得自动恢复

### F4. 转发侧租户账号与 API Key 同步

**优先级**：P0

Go 转发侧 token 归属于 user。模型平台以租户作为治理边界，一个模型平台租户对应一个转发侧租户账号。

#### 产品规则

1. 租户首次创建大模型接入凭证时按需创建转发侧租户账号
2. 模型平台保存 `tenant_id → relay_user_id` 映射
3. 同租户后续 API Key 复用同一个 `relay_user_id`
4. 转发侧租户账号是技术账号，不等同于模型平台真实用户
5. API Key 创建人、权限、离职状态和审计仍由模型平台维护
6. API Key 状态或授权变化后模型平台主动同步 Go 转发侧 token
7. 同步完成或失败后都要更新模型平台侧同步状态
8. 同步后主动清理 Redis 缓存，不等待自然过期
9. 全部资源权限的 API Key 在 Go 侧不得固化模型或接入点白名单

#### 同步场景

| 触发         | 动作                          |
| ---------- | --------------------------- |
| 创建 API Key | 创建或复用转发侧租户账号，创建 Go token    |
| 禁用 API Key | Go token 置为不可用，清理缓存         |
| 启用 API Key | Go token 置为可用，清理缓存          |
| 删除 API Key | 删除或禁用 Go token，清理缓存         |
| 权限范围变化     | 更新 Go token 有效权限，清理缓存       |
| 资源关闭失效绑定产生 | 从 Go token 有效权限中移除对应目标，清理缓存 |
| 资源关闭失效绑定恢复 | 将对应目标加入 Go token 有效权限，清理缓存  |
| 离职资源清理     | 同步禁用或删除状态，清理缓存              |

#### 验收标准

1. 首次创建凭证时自动创建转发侧租户账号
2. 同租户后续复用同一个 `relay_user_id`
3. 禁用后转发侧调用立即失败
4. 删除后 Go token 和 Redis 缓存均被清理
5. 失效绑定产生后 Go token 和缓存被更新
6. 失效绑定恢复后 Go token 和缓存被更新
7. 全部资源权限的 API Key 新增启用资源后不需要逐把更新 Go token
8. 同步失败时详情展示同步失败状态供排查

### F5. 调用日志事实流

**优先级**：P0

调用明细事件是调用日志、用量统计、告警、审计和 OM 计量的事实来源。使用 Kafka 承接调用明细事件，ES 只作为前台查询索引。

#### 产品规则

1. 每次调用尝试都必须有可追踪记录
2. 调用明细事件优先写入 Kafka
3. Kafka message key 使用 `tenant_id:api_key_id`
4. 事件使用 `event_id` 或 `request_id` 做幂等
5. ES 消费者写入月度索引（如 `call_log_202606`）
6. 前台查询按时间范围命中一个或多个 ES 月度索引
7. 用量消费者基于调用明细事件更新统计表
8. 告警消费者基于调用明细事件处理异常和安全护栏提醒
9. OM 计量消费者基于调用明细事件提交或扣减点数

#### 事件覆盖范围

| 场景               | 是否记录                   |
| ---------------- | ---------------------- |
| 调用成功             | ✅ 记录实际命中模型/Token/点数/耗时 |
| 上游失败最终失败         | ✅ 记录错误码/候选尝试和 retry 结果 |
| 路由失败             | ✅ 记录接入点/候选不可用原因        |
| AI 安全护栏观察命中      | ✅ 调用继续，记录风险结果          |
| AI 安全护栏拦截        | ✅ 调用被阻断，记录拦截原因         |
| 余额不足 | ✅ 未请求上游，记录前置校验失败（本期存在此场景：后台配置限额后触发余额不足的前置校验） |
| 无调用资格            | ✅ 未请求上游，记录前置校验失败       |
| API Key 无效或无授权   | ✅ 未请求上游，记录前置校验失败       |

#### 事件核心字段

`event_id`/`request_id`、`tenant_id`/`user_id`/`api_key_id`、`target_type`/`target_id`、`endpoint_id`/`endpoint_type`、`modelId`/`providerId`、`status`/`error_code`/`error_message`、`retry_result`、`tokens_input`/`tokens_output`、`points_consumed`、`guardrail_result`、`latency_ms`/`ttft_ms`、`created_at`

#### 验收标准

1. 调用成功后可通过 Kafka 查到对应事件
2. 前置校验失败时不请求上游但仍可查到失败记录
3. ES 消费失败后可通过 Kafka 重放补写（失败指 ES 消费者写入 ES 失败；重放需人工通过运维工具触发 Kafka consumer offset 重置；重放期间前台查询可能暂不包含未补写数据，补写完成后自动恢复完整）
4. 跨月查询能命中多个 ES 月度索引
5. 同一事件重复消费不造成重复统计或扣减
6. 日志详情展示实际命中模型、供应商来源、Token、点数、耗时、安全护栏结果和错误信息

#### 待确认项

- T4：Kafka topic 分区数、保留周期、Schema 管理方式
- T5：ES 月度索引模板、读别名、生命周期策略

### F6. 用量视角、**调用资格**与用量限额

**优先级**：P0

不同角色看到不同范围的用量数据。调用资格由本地状态判断（OM 点数包、试用、白名单）。用户级月度 Token 用量限额用于控制异常成本。

#### 用量视角

| 角色    | 默认视角  | 可切换       |
| ----- | ----- | --------- |
| 普通用户  | 我的用量  | 不可切换      |
| 租户管理员 | 本租户用量 | 我的用量      |
| 平台管理员 | 全平台用量 | 指定租户、我的用量 |

> 页面**必须**明确展示当前视角

#### 统计维度

时间范围、租户、用户、API Key、接入点、实际命中模型、供应商来源、调用状态、Token 用量、点数消耗、失败率

> 公共接入点调用命中租户未直接启用模型时，用量统计仍可按实际命中模型筛选和聚合，但不提供跳转到未启用模型详情或绑定操作。

#### 调用资格规则

1. 具备以下**任一**条件视为有调用资格：已购买有效 OM 点数包、处于试用期、命中平台维护的调用白名单。**已澄清：调用资格本期不做**，此规则推迟至后续迭代，本期模型平台不实现调用资格的前置校验逻辑
2. 没有调用资格时禁止调用生产模型
3. 点数不足时调用前置校验失败
4. OM 状态通过事件/定时任务/低频 check 同步到本地，主链路不在每次调用时远程查询
5. OM 消费 Kafka 或扣减失败，不回滚已返回给调用方的模型响应；后续通过重试/对账/运营处理

#### 错误提示

| 场景                  | 提示                                  |
| ------------------- | ----------------------------------- |
| 无调用资格               | 当前租户暂无模型调用资格，请购买 OM 点数包或联系平台管理员     |
| 点数不足                | 当前租户点数余额不足，暂时无法调用生产模型               |
| OM 同步异常             | 调用资格状态暂时无法确认，请稍后重试或联系平台管理员          |
| 用户月度超额 | 当前用户本月模型调用额度已用尽，请等待下月恢复或联系平台管理员调整额度（额度通过后台配置表或运维方式设置，测试时需开发提供对应测试数据） |

#### 用户级月度 Token 限额

**短解（快速止血）**：

1. 限额主体是**用户**，不是单 API Key。用户多个 API Key 的 Token 合并计入
2. 限额归属用户沿用 API Key 创建人，不识别真实终端使用人
3. 只有进入**限额租户白名单**的租户才启用限额治理    
4. 租户启用后：已配置用户按配置额度阻断；未配置用户默认不限额，但用量仍进入统计/审计/治理分析
5. 额度为 0 表示显式禁止该用户继续产生受限范围内的模型调用成本；删除配置表示回到未配置用户规则
6. 短解不新增完整配置页面，平台管理员通过**后台配置表或运维方式**维护白名单和用户限额。**测试数据依赖**：需提前与开发约定限额白名单、用户额度配置的测试数据提供方式和时间节点
7. 按自然月统计；新自然月已用 Token 归零，额度配置沿用
8. 用户达限额时调用前置校验拒绝，返回**不可重试失败**，不触发接入点路由切换
9. 因模型输出 Token 只能在调用完成后确认，允许最后一次已放行调用轻微超过限额

**长解（全平台成本治理演进）**：

> **已澄清：长解本期不做**，以下内容保留作为演进方向参考，不进入 S4 开发与测试范围。

1. 通过**模型成本归属**区分平台成本模型和客户自有模型；平台成本模型纳入用户级月度 Token 限额，客户自有模型不纳入
2. 直接调用模型和通过接入点调用都遵循**入口无关成本限额**；命中平台成本模型就计入同一限额
3. 接入点场景支持**路由后限额校验**：先完成 API Key/资格/授权/接入点状态校验，再根据实际命中模型的成本归属判断是否限额
4. 长解首版优先实现月度用量限额，RPM/TPM 调用限流只保留策略设计和扩展位置

#### 限额数据与一致性

1. 调用前置校验不扫描日志或查询明细；短解使用 **Redis 月度用户 Token 计数**作主链路快速判断
2. Redis 不是限额用量唯一事实源；需持久化**月度用户 Token 账本**用于恢复/对账/运维查询
3. 账本按自然月+租户+用户累计，不按 API Key 分配额度
4. 限额用量更新必须以调用明细事件 ID 或 `request_id` 做幂等，避免重复累计
5. Redis 计数与账本不一致时以持久化账本为准重建或修正
6. Redis 不可用时默认**放行**并记录异常，恢复后通过账本修正
7. 限额配置必须持久化，含配置主体/额度/启停状态/操作人/操作时间
8. 限额配置变更后下次前置校验生效，不依赖缓存自然过期
9. 同一模型调用只能命中**一套**成本限额策略，避免短解长解重复阻断

#### 验收标准

1. 未购买 OM 点数包且无试用/白名单的租户无法调用生产模型
2. 点数不足时请求被拒绝并返回明确错误
3. 普通用户只能查看"我的用量"
4. 租户管理员默认查看"本租户用量"，可切换到"我的用量"
5. 平台管理员默认查看"全平台用量"，可切换到指定租户或我的用量
6. 通过公共接入点命中未直接启用模型后，用量统计可按该实际命中模型筛选
7. 未启用模型筛选项不提供模型详情跳转、绑定或创建模型级 API Key 操作
8. 进入限额白名单且已配置额度的用户，月度 Token 用量达标后后续调用被前置拒绝
9. 同一用户多API Key 共享同一个用户级月度 Token 限额
10. 未配置用户默认不被限额阻断，但其用量仍进入统计和审计
11. 限额用量累计以调用明细事件 ID 或 `request_id` 幂等
12. Redis 不可用时默认放行并记录异常，后续通过账本恢复和对账

### F7. 离职资源清理

**优先级**：P0

基于 IAM 离职人员名单识别 API Key，提供租户管理员清理入口。

#### 产品规则

1. 离职人员由 IAM 识别，平台不自行判定
2. 通过定时任务同步离职人员名单（接口 `POST /CRM_WebApi/api/GetCMSMV`，`Isresign=N`），**每天同步一次；本期不支持手动触发同步，不支持自定义同步时间**
3. 离职治理能力受**部署级离职治理启用配置**控制；关闭时不启动同步、不展示入口、不访问相关表
4. API Key 是否属于离职资源以**创建人**为准
5. 默认策略：**禁用**离职人员创建的 API Key
6. 特定 AI Coding 租户可配置为**自动删除**（通过**后台配置表**维护，配置维度为租户级别，支持动态修改；默认策略为禁用，仅白名单内的租户启用自动删除）
7. 离职清理是凭证级处置，**不产生**资源关闭失效绑定
8. 租户管理员可一键删除待处理 API Key
9. 自动和手动处置均进入审计日志（**本期审计日志仅在数据库/后台可查，不提供前端查询页面**）
10. 本期**不发送真实离职通知**，仅保留通知状态和页面提示

#### 状态流转

正常 → 离职待处理 → 已禁用待删除/自动删除中 → 已删除/处置失败

#### 验收标准

1. IAM 返回离职人员后按 API Key 创建人识别
2. 部署级配置关闭时不启动同步、不展示入口
3. 离职 API Key 即使仍被业务系统使用也进入清理流程
4. 未配置自动删除策略的租户仅禁用不自动删除
5. 配置自定义删除策略的 AI Coding 租户自动删除
6. 清理后不新增资源关闭失效绑定
7. 租户管理员可一键删除本租户待处理 API Key
8. 禁用/删除/同步失败均进入审计日志
9. 本期不发送真实离职通知

### F9. 模型发布与默认启用策略

**优先级**：P1

纳管模型上架或公共接入点分配给租户时，需支持设置资源进入租户平台资源池后的默认启用策略（平台管理员配置租户资源池）。

#### 发布流程

1. 平台管理员确认纳管模型信息
2. 选择模型进入哪些租户的平台资源池，设置默认启用状态
3. 选择是否生成公共接入点（默认不勾选）
4. 如勾选，配置接入点名称、路由策略、状态、加入租户资源池范围和默认启用状态
5. 可选套用 AI 安全护栏推荐配置（默认不启用，启用后默认观察模式）
6. 提交→资源进入对应租户的平台资源池

#### 产品规则

1. 纳管模型发布前必须明确是否加入租户平台资源池
2. 加入时必须设置默认启用状态
3. 默认启用策略只影响首次进入，不覆盖租户管理员手动配置
4. 生成公共接入点必须由平台管理员显式确认
5. 未勾选不自动创建接入点
6. 公共接入点生成后可独立加入或移出租户资源池
7. 公共接入点进入租户资源池不要求租户同时启用背后候选模型

#### 验收标准

1. 上架后必须明确是否加入资源池
2. 可设置默认启用或默认关闭
3. 租户管理员手动关闭后平台更新同一资源不得重新打开
4. 未勾选不自动创建接入点
5. 勾选后创建公共接入点并带入初始配置
6. 公共接入点可独立加入或移出资源池
7. AI 安全护栏推荐配置不默认启用；启用后默认观察模式

### F10. `/v1/responses` 无状态透传

**优先级**：P1

为避免多节点部署下对上下文存储和 sticky session 产生错误预期，本期明确只做无状态透传。

#### 产品规则

1. 平台不存储 Responses 协议上下文
2. 原样转发 `previous_response_id`、`input`、`instructions` 等字段
3. 多节点部署不要求 sticky session
4. 平台只记录调用日志、用量和审计
5. 不保证 `previous_response_id` 跨供应商/跨模型/跨接入点可用
6. 上下文连续性由上游模型供应商协议能力决定

#### 验收标准

1. 多节点请求落到不同节点不影响转发
2. `previous_response_id` 原样转发
3. 未传上下文字段时平台不自行补充
4. 跨供应商/模型上下文不可用时按上游错误返回

### F11. 模型供应商 MODEL\_ID 变更影响治理

**优先级**：P0

接入点路由候选绑定的是具体纳管模型，绑定键按 `providerId + modelId` 理解。平台管理员编辑供应商 MODEL\_ID 等字段可能导致接入点原候选无法继续路由到可用上游。

#### 产品规则

1. 接入点候选模型绑定以 `providerId + modelId` 为准，不因供应商或渠道配置变更自动切换
2. 编辑 MODEL\_ID、模型列表、模型映射、上游地址前，系统应提供**影响分析**
3. 影响分析必须覆盖公共接入点和业务接入点
4. 影响分析至少展示：受影响接入点数量、ID、名称、类型、候选位置、路由角色、优先级、权重、归属租户、当前状态
5. 候选变更后不可用时接入点进入需重新绑定或异常状态；不产生 API Key 资源关闭失效绑定
6. 同一候选后续恢复可用时接入点可继续调用，不要求重建 API Key
7. 切换供应商/模型/渠道必须编辑接入点路由候选重新同步
8. 接入点编辑页**必须**支持将受影响候选直接替换为新的 `providerId + modelId`，一次保存完成删除旧映射→写入新映射→同步转发侧→清理缓存
9. 供应商/渠道变更确认页只提示影响范围，**不提供**批量重绑入口
10. 直接重新绑定时应保留候选在路由中的位置、角色、优先级和权重
11. 公共接入点由平台管理员处理影响；业务接入点让创建租户管理员看到候选异常并自行编辑恢复
12. 本期不做业务接入点候选异常的**主动通知**

#### 页面与交互

**供应商/渠道编辑提交影响字段变更时**：

- 变更字段：MODEL\_ID、模型列表、模型映射、上游地址
- 影响对象：受影响公共接入点和业务接入点列表
- 影响说明：不会自动切换；调用可能失败；恢复同一候选可继续
- 操作选项：取消、继续保存
- 不提供批量重绑入口

**接入点详情候选异常展示**：

- 候选模型不可用或上游解析失败
- 最近失败时间和原因
- 建议操作

**接入点列表候选异常摘要**：

- 状态：候选正常/部分候选异常/全部候选异常
- 异常候选数量、最近异常时间

**接入点编辑页候选直接重新绑定**：

- 在原候选行点击重新绑定
- 选择新 `providerId + modelId`
- 保存→替换→同步转发侧→清理缓存

#### 验收标准

1. 编辑影响字段前系统展示受影响接入点列表
2. 列表同时覆盖公共接入点和业务接入点
3. 确认保存后不自动切换到其他候选
4. 不可用候选不产生 API Key 资源关闭失效绑定
5. 不可用候选详情展示需重新绑定或异常状态
6. 编辑页支持旧候选直接替换为新候选
7. 确认页不提供批量重绑入口
8. 直接绑定保留原路由位置/角色/优先级/权重
9. 同一候选恢复后可继续调用
10. 手动编辑后重新同步转发侧并清理缓存
11. 业务接入点候选异常时创建租户管理员可在列表和详情看到
12. 本期不主动通知租户管理员

## 9. 非功能需求

| 类别  | 要求                                                 |
| --- | -------------------------------------------------- |
| 可用性 | 调用前置校验基于本地状态或缓存；OM、IAM、供应商健康不在每次调用时远程查询            |
| 一致性 | API Key 状态、权限范围、资源关闭失效绑定和离职清理必须主动同步转发侧并清理 Redis 缓存 |
| 可审计 | 平台资源池、租户资源启用配置、API Key 状态变化、离职清理、转发侧同步均记录审计日志      |
| 可扩展 | Kafka 消费者可独立扩展，ES、用量、告警、OM 计量互不阻塞                  |
| 可恢复 | ES 写入失败后可通过 Kafka 重放；消费者按事件 ID 幂等                  |
| 可运维 | ES 月度索引支持跨月查询、保留策略和归档                              |
| 安全  | 不展示供应商密钥、内部转发账号、未启用模型操作入口；API Key 仅创建时展示一次         |
| 兼容性 | 保留现有基础调用链路，迁移存量 API Key 时需提供兼容或迁移方案                |

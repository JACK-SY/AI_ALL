## 1. 一句话说明

Sprint4 要把模型平台从“可以调用模型”升级为“可授权、可治理、可审计、可运营的模型调用平台”。

本期重点不是新增模型训练能力，而是补齐模型被纳管后如何被租户发现、授权、调用、记录、统计和清理的闭环。

***

## 2. 背景

当前平台已经具备模型广场和基础 API Key 调用能力，但治理能力不足：

| 问题           | 现状影响                                                                            |
| ------------ | ------------------------------------------------------------------------------- |
| 模型默认可见边界不清   | 模型上架后缺少租户级授权控制，平台管理员无法精确控制哪些租户可见、可绑定、可调用                                        |
| 接入点职责混在一起    | 平台维护的公共入口、租户自建的业务入口和租户资源开关没有清晰边界，路由配置、资源启用状态和维护责任容易混淆                           |
| API Key 治理不足 | 大模型、小模型凭证混杂；权限粒度不足；资源关闭、人员离职后缺少闭环处置                                             |
| 调用日志事实源薄弱    | 调用日志直接依赖查询存储，缺少可重放、多消费者的事实流，影响排障、审计、用量和计费核对                                     |
| 用量视角不清       | 不同角色看到的用量范围不清；调用资格判断（OM 点数包、试用、白名单联动）本期不做，推迟到后续迭代                               |
| 用户级成本限额缺失    | AI Coding 等高消耗场景缺少按用户月度 Token 控制能力，异常个人使用可能放大全租户或平台成本                           |
| 同名模型无法精确调用   | 同一 `modelId`（如 `glm-5.1`）来自多个模型供应商时，OpenAI 协议的 `model` 参数无法指定供应商，网关按模型名在多个渠道中选择 |
| 调用日志单索引持续膨胀  | `kmo_call_log` 单索引每月约 400GB（请求 / 响应原文占绝对大头），查询、滚动和清理都受制约                        |

## 3. 产品目标

| 编号 | 目标                   | 成功标准                                                                                    |
| -- | -------------------- | --------------------------------------------------------------------------------------- |
| G1 | 租户授权范围边界清晰           | 平台可通过租户授权范围（白名单 + 黑名单叠加）给出租户可用资源边界；授权即可见可调用，无租户侧启停环节；租户资源启用配置层与默认启用策略移出本期（Backlog A-10） |
| G2 | 接入点边界清晰且可精确寻址        | 公共接入点由平台维护，业务接入点由租户维护；接入点候选为渠道锁定候选，同名 `modelId` 来自不同供应商时可通过指定模型接入点精确调用                  |
| G3 | API Key 可治理、好用       | 列表可按创建人筛选；明文仅限创建人取回并记录审计；凭证可一键填入 CC Switch                                              |
| G4 | 离职资源可清理              | 平台能识别离职人员创建的 API Key，并按租户策略禁用或删除                                                        |
| G5 | 调用事实可追溯且存储可持续        | 调用日志查询索引与内容索引分离，按周期滚动并可整索引过期，单索引无限膨胀问题解除；Kafka 事实流推迟到后续迭代                               |
| G6 | 用量视角和用户级 Token 限额可运营 | 普通用户、租户管理员、平台管理员看到不同范围的用量；平台可对指定租户下的用户按月度 Token 用量进行限额治理；调用资格联动（OM 点数包、试用、白名单）本期不做      |

## 4. 非目标

本期明确不做以下内容（完整清单与 Backlog 归属见 §4a）：

- 模型训练、训练任务、训练数据集、训练产物、微调流程。
- 用户级模型开通流程（申请接入、审批、允许名单、撤销开通），不含本期用户级用量限额。
- 平台托管 `/v1/responses` 上下文状态。
- AI 安全护栏独立一级菜单和复杂策略编排。
- 资源关闭失效绑定的手动重新生效、重新选择资源或批量清理入口。

## 6. 核心用户场景

### 场景 1：平台授权租户可用资源

平台把公共接入点和公共模型授权给某租户（进入租户授权范围）后，租户内用户即可在模型广场、接入点列表和 API Key 绑定选择器中看到并使用这些资源（授权即生效）。

### 场景 2：租户通过公共接入点调用模型

平台管理员创建公共接入点，并把该公共接入点授权给租户。授权后租户用户即可通过它调用模型，不需要背后候选模型对该租户单独授权；租户只能看到路由摘要，不能编辑平台路由配置。

### 场景 3：租户创建自己的业务接入点

租户管理员创建业务接入点，用于本租户业务系统调用。业务接入点只能选择本租户授权范围内（未被封禁）的模型作为候选。其他租户不可见、不可绑定、不可调用该业务接入点。

### 场景 4：开发者创建大模型接入凭证

开发者在“大模型接入凭证”中创建 API Key，可以选择全部资源权限，也可以只绑定指定模型或接入点。API Key 可同时绑定模型和接入点。选择器只展示本租户授权范围内的资源；调用目标以 `ep-*` 开头时按接入点授权和 API Key 权限校验，否则按模型授权和 API Key 权限校验。

### 场景 5：平台撤权或封禁后保留失效绑定

平台撤销某租户对某个模型或公共接入点的授权，或把该租户加入该资源黑名单后，已绑定该资源的 API Key 不被删除，API Key 也不整体禁用。对应绑定进入失效状态（租户侧统一展示「平台撤销授权」），后续调用立即拒绝；同一 API Key 的其他有效绑定继续可用。平台重新授权或解封后，历史绑定自动恢复有效。

### 场景 6：离职人员 API Key 自动清理

IAM 同步离职人员名单后，模型平台定时任务自动识别离职人员创建的 API Key 并直接删除，无需租户管理员手动确认。

### 场景 7：平台通过调用日志排障和审计

每次调用结束后生成调用日志：监控与检索字段写入查询索引，请求与响应原文写入内容索引，两者用调用内容引用（`content_ref`）关联。平台管理员可通过 `request_id` 查询调用过程、实际命中模型、Token、点数、安全护栏结果和错误信息，必要时经 `content_ref` 受控追溯原文。

### 场景 8：平台限制用户月度 Token 用量

平台管理员维护「平台限额启用租户名单」并为具体用户配置额度（AI Coding 等高消耗场景是典型用例）。用户通过自己名下任意 API Key 调用模型时，平台按 API Key 创建人汇总该用户当月 Token 用量（仅 `status=SUCCESS` 调用累加）；达到额度后，调用前置校验拒绝继续调用。启用租户内未单独配额度的用户走平台默认额度；不在启用租户名单内的租户不受限额约束，但用量仍进入统计、审计和后续治理分析。

### 场景 9：开发者精确调用指定供应商的同名模型

阿里云和百度千帆都提供 `glm-5.1`。平台管理员或租户管理员创建指定模型接入点（如 `ep-glm-ali`），候选渠道锁定到阿里云 `glm-5.1`。开发者通过 OpenAI 协议以 `model=ep-glm-ali` 调用，网关直达该渠道，不再按模型名在多个渠道间选择。

### 场景 10：开发者把凭证一键填入 CC Switch

开发者在大模型接入凭证列表的行操作中点击“填入 CC Switch”，选择应用（Claude / Codex / Gemini）和模型（可选 `ep-*` 接入点），平台生成 `ccswitch://` 深链拉起本地 cc-switch 完成 provider 导入。明文取回仅限创建人本人，动作记录审计日志。

### 场景 11：平台按黑白名单管理公共资源可见范围

平台管理员发布一个内部实验模型，把可见范围模式设为 `WHITELIST`，只授权给内部租户。未授权租户在租户侧完全看不到该资源（列表、筛选、选择器均不出现），不可绑定，直调返回 `RESOURCE_NOT_AUTHORIZED`。某个白名单内租户滥用该资源时，平台把该租户加入资源黑名单（黑名单优先级高于白名单，白名单条目保留）：封禁期间该租户不可见、不可调用，已绑定 API Key 展示「平台撤销授权」失效原因；解封后白名单资格与失效绑定自动恢复。`ALL` 模式资源同样可用黑名单排除个别租户，无需整体切为 `WHITELIST`。

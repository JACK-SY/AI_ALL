# F11. 模型供应商 MODEL_ID 变更影响治理

> 状态：✅ 已完成（上一迭代交付）。以下内容作为已实现行为基线保留。

## 8.11.1 需求说明

接入点路由候选绑定的是具体纳管模型，绑定键按 `providerId + modelId` 理解。实际调用时，平台根据候选模型关联的模型供应商或渠道配置，解析到对应的上游地址和上游模型 ID。平台管理员编辑模型供应商或渠道下的 MODEL_ID、模型列表、模型映射或上游地址时，可能导致接入点原候选无法继续路由到可用上游。

本需求要求平台在变更前提示影响范围，变更后不自动把接入点候选切换到其他模型或供应商，避免同名模型误切、成本策略变化或权限边界漂移。同时，接入点编辑页必须支持把受影响候选直接重新绑定到新的 `providerId + modelId`，不要求用户先临时切换到第三个模型。

## 8.11.2 产品规则

- 接入点候选模型绑定以 `providerId + modelId` 为准，不因模型供应商或渠道配置变更自动切换到另一个模型。
- 编辑模型供应商或渠道的 MODEL_ID、模型列表、模型映射、上游地址前，系统应提供影响分析。
- 影响分析必须覆盖公共接入点和业务接入点。
- 影响分析至少展示受影响接入点数量、接入点 ID、接入点名称、接入点类型、候选模型位置、路由角色、优先级、权重、归属租户和当前状态。
- 如果该候选模型变更后无法解析到可用上游，接入点候选进入需重新绑定或异常状态，调用按路由策略失败或切换到其他有效候选；不产生 API Key 资源关闭失效绑定。
- 如果同一个候选后续恢复可用，接入点可继续调用，不需要重建 API Key。
- 如果平台管理员希望切换到其他模型、其他供应商或其他渠道，必须编辑接入点路由候选并重新同步转发侧映射。
- 接入点编辑页必须支持将受影响候选直接替换为新的 `providerId + modelId`，一次保存完成删除旧映射、写入新映射、同步转发侧并清理缓存。
- 模型供应商或渠道变更影响确认页只提示影响范围，不提供批量重绑入口。
- 直接重新绑定时，应保留候选在路由中的位置、路由角色、优先级和权重，除非用户明确修改。
- 公共接入点由平台管理员处理影响；业务接入点应让创建租户管理员在接入点列表和详情看到候选异常，并自行编辑恢复。
- 变更、影响确认、接入点候选异常和恢复均应记录审计日志。

## 8.11.3 边界条件与异常场景

- **候选恢复后自动化验证需手动刷新**：候选模型恢复可用后，接入点详情页不会自动刷新状态，需用户手动刷新页面或重新进入详情才能看到状态变更。
- **公共业务交叉场景影响**：同一候选模型同时被公共接入点和业务接入点绑定时，影响分析须合并展示，避免重复计数接入点数量；影响确认页分别标注公共接入点与业务接入点数量。
- **并发编辑竞态**：多个管理员同时编辑同一个模型供应商的影响字段时，采用后写入覆盖策略并写审计日志，不做行级锁。
- **大批量影响场景**：受影响接入点超过 100 个时，影响确认页默认折叠列表仅展示数量和前 10 条，用户点击"展开全部"查看完整列表。
- **候选重新绑定期间调用**：重新绑定保存操作期间到达的调用，按旧候选路由；保存完成后新映射生效，中间不丢失调用。
- **级联影响**：渠道变更影响传播到供应商级别时，影响分析仅立足于直接绑定该候选的接入点，不做多层级联追溯。

## 8.11.5 页面交互

### 模型供应商或渠道编辑页 — 影响确认二次确认框

- 触发条件：提交 MODEL_ID、模型列表、模型映射、上游地址字段的变更。
- 弹窗内容：变更字段摘要 → 受影响接入点数量（分别标注公共 / 业务） → 受影响接入点列表 → 影响说明（接入点不会自动切换；变更后调用可能失败，恢复同一候选后可继续调用）。
- 操作按钮：取消、继续保存。
- 大批量折叠：受影响接入点超过 100 个时，列表默认折叠仅展示前 10 条，底部显示"展开全部（N 个）"按钮。
- 不提供批量重新绑定入口。

### 接入点详情页 — 候选异常展示

- 候选行增加异常图标与异常标签（候选异常 / 上游不可用）。
- 展开候选行可查看最近失败时间与失败原因。
- 底部建议操作区：等待同一供应商/渠道恢复（静态文案）/ 重新绑定（跳转编辑页）。
- 恢复可用后需手动刷新页面查看状态变更。

### 接入点列表页 — 候选异常摘要列

- 新增"候选状态"列，标签式展示：正常（绿）/ 部分异常（黄）/ 全部异常（红）。
- 异常标签旁展示异常候选数量与最近异常时间。

### 接入点编辑页 — 候选直接重新绑定

- 在原候选行点击"重新绑定"按钮（仅异常候选行展示该按钮，正常候选行不展示）。
- 弹出候选选择器，选择新的 `providerId + modelId`。
- 保存后替换原候选，同步转发侧映射并清理缓存。
- 保存操作期间到达的调用按旧候选路由；保存完成后新映射生效。

## 8.11.6 状态流转

| 候选状态 | 触发条件 | 目标状态 | 说明 |
|----------|----------|----------|------|
| 正常 | 供应商/渠道 MODEL_ID 变更导致上游不可解析 | 候选异常 | 按路由策略失败或切换其他有效候选 |
| 候选异常 | 同一候选上游恢复可用 | 正常 | 需手动刷新页面查看 |
| 候选异常 | 管理员直接重新绑定新候选 | 正常 | 保留原路由位置/角色/优先级/权重 |
| 正常 | 管理员手动替换候选 | 正常 | 写入新映射、同步转发侧、清理缓存 |

## 8.11.7 验收标准

### 一、变更影响分析（AC-11.1 ~ AC-11.3）

- [ ] **AC-11.1** Given 管理员编辑模型供应商或渠道的 MODEL_ID / 模型列表 / 模型映射 / 上游地址；When 提交变更；Then 系统在保存前展示受影响接入点列表，包含接入点 ID、名称、类型、候选位置、路由角色、优先级、权重、归属租户、当前状态。
- [ ] **AC-11.2** Given 变更影响同时绑定公共接入点和业务接入点的候选；When 展示影响分析；Then 列表同时包含公共接入点和业务接入点，分别标注数量。
- [ ] **AC-11.3** Given 管理员确认保存变更；When 变更生效后；Then 系统不自动把受影响接入点切换到其他模型、其他供应商或其他渠道。

### 二、异常处理与恢复（AC-11.4 ~ AC-11.9）

- [ ] **AC-11.4** Given 受影响候选不可用；When 接入点收到调用；Then 按路由策略失败或切换到其他有效候选。
- [ ] **AC-11.5** Given 受影响候选不可用；When 候选进入异常状态；Then 不产生 API Key 资源关闭失效绑定。
- [ ] **AC-11.6** Given 接入点候选进入异常状态；When 管理员查看接入点详情；Then 详情页展示候选异常标记、最近失败时间与失败原因。
- [ ] **AC-11.7** Given 接入点候选处于异常状态；When 管理员在编辑页点击"重新绑定"并选择新候选保存；Then 系统删除旧映射、写入新映射、同步转发侧并清理缓存，保留原路由位置/角色/优先级/权重。
- [ ] **AC-11.8** Given 管理员在供应商/渠道变更影响确认页；When 查看影响列表；Then 页面仅提供"取消"和"继续保存"，不提供批量重新绑定入口。
- [ ] **AC-11.9** Given 同一候选上游恢复可用；When 接入点收到调用；Then 可正常调用，无需重建 API Key。

### 三、交互与通知（AC-11.10 ~ AC-11.12）

- [ ] **AC-11.10** Given 受影响接入点超过 100 个；When 展示影响确认弹窗；Then 列表默认折叠仅展示前 10 条，底部提供"展开全部"按钮。
- [ ] **AC-11.11** Given 业务接入点候选异常；When 创建租户管理员查看接入点列表或详情；Then 可看到异常状态和建议操作。
- [ ] **AC-11.12** Given 业务接入点候选异常；When 异常发生；Then 本期不向租户管理员主动发送通知。

### 四、审计与并发（AC-11.13 ~ AC-11.14）

- [ ] **AC-11.13** Given 供应商/渠道变更、影响确认、候选异常、恢复、重新绑定操作发生；When 任一操作完成；Then 系统记录审计日志。
- [ ] **AC-11.14** Given 多个管理员同时编辑同一供应商影响字段；When 并发提交；Then 采用后写入覆盖策略并写审计日志。

## 8.11.8 补充约束

- 影响分析查询性能需满足：1000 个接入点场景下影响分析查询响应时间不超过 3 秒。
- 接入点候选重新绑定操作需保证幂等性，重复提交同一变更产生一致结果。
- 候选异常状态需在数据库中持久化存储，重启后保留状态不丢失。

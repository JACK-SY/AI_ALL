# F1. 平台资源池与租户授权范围

## 8.1.1 需求说明

平台把公共资源（模型、公共接入点）授权给租户，形成租户授权范围；资源进入租户授权范围即对租户内用户可见、可绑定、可调用（授权即生效），可见性完全由平台侧维护。普通用户不直接理解平台资源池、租户授权范围或黑白名单，只看到过滤后的结果。

平台需要对公共模型和公共接入点做租户维度的可见性控制，覆盖场景包括：新模型只给公测租户灰度、内部实验模型只给内部租户、商务付费分层、个别滥用租户快速止损。现状审计确认：当前代码里 `visibility=PUBLIC` 等价于「全部租户可见」，无任何租户过滤面。

本节定义平台侧的「租户 × 公共资源」维护面：

| 层                            | 维护人   | 本期状态                             |
| ---------------------------- | ----- | -------------------------------- |
| 平台资源池（universe）              | 平台管理员 | 平台已纳管、可作为公共资源分发的全部公共接入点和公共模型全集   |
| 租户授权范围（per-tenant，白名单 + 黑名单） | 平台管理员 | 本节的管理对象；本期唯一的租户 × 资源控制面，授权即可见可调用 |

私有模型和业务接入点的可见性归属租户自身，不属于本节范围；用户级可见性通过 API Key 绑定层解决。

### 资源类型

| 资源类型  | 进入租户授权范围的来源                                         | 授权后效果                                      |
| ----- | --------------------------------------------------- | ------------------------------------------ |
| 模型    | 平台授权给租户的公共模型（租户授权范围的模型部分）；首版可由该租户可用公共接入点绑定的候选模型列表形成 | 租户内用户可在模型广场看到、创建模型级 API Key、模型直调、作为业务接入点候选 |
| 公共接入点 | 平台创建公共接入点并授权给租户                                     | 租户内用户可在接入点列表看到、绑定 API Key、通过 `ep-*` 调用     |

模型和公共接入点是两类独立资源。撤销或封禁公共接入点授权不联动其背后候选模型的授权；撤销或封禁模型授权不影响已授权公共接入点的内部路由。

### 触发条件

- 平台管理员纳管新公共模型或创建新公共接入点
- 平台管理员在资源授权范围页中执行租户授权/租户撤权/黑名单/模式切换操作
- 模型/接入点状态变更触发资源授权列表数据联动

### 前置依赖

- 目标公共资源已完成纳管且状态可用
- 操作人具备平台管理员角色权限
- 租户已在平台创建并激活

### 涉及角色

| 角色    | 职责                                       |
| ----- | ---------------------------------------- |
| 平台管理员 | 维护模型库、配置资源授权范围（白名单/黑名单/模式切换）、创建公共接入点     |
| 租户管理员 | **无资源授权范围菜单入口**                          |
| 普通用户  | 消费授权范围内的资源（模型广场浏览、API Key 绑定、调用），无授权管理入口 |

### 映射产品目标

| 产品目标编号 | 映射说明                                         |
| ------ | -------------------------------------------- |
| G1     | 租户资源启用边界清晰 — 本功能直接交付租户授权范围作为平台侧资源边界，授权即可见可调用 |
| G2     | 接入点边界清晰且可精确寻址 — 公共接入点授权独立于模型授权，接入点维度边界由本功能控制 |

## 8.1.2 产品规则

- 租户授权范围限制租户内用户可见可用的公共资源边界；授权即生效，本期无租户侧启停环节。
- 每个公共资源（模型按 `providerId + modelId`，公共接入点按 `ep-xxx`）有一个可见范围模式 `visibility_mode`，并可叠加黑名单条目。
- 可见性判定公式：`可见(租户, 资源) = 租户不在该资源黑名单 AND（visibility_mode = ALL 或 租户在该资源白名单）`。
- `ALL`（缺省模式）：对全部租户默认可见，自动进入所有租户的授权范围，无需逐租户配置。
- `WHITELIST`（模式）：仅对被平台显式授权的租户可见。
- 黑名单（否决条目，非模式取值）：优先级最高，被列入的租户对该资源不可见、不可调用，无论模式与白名单如何。
- 黑名单只在 `ALL` 模式下有实际意义：`WHITELIST` 模式下未授权租户本身已不可见，无需额外加黑名单。
- 白名单条目与黑名单条目共存不互斥：封禁 = 新增黑名单条目（白名单条目原地保留）；解封 = 删除黑名单条目，临时封禁不破坏授权事实。
- 黑名单条目带可选封禁原因（`reason`）。
- 模型授权与公共接入点授权相互独立：撤销或封禁公共接入点授权只影响该接入点在该租户下的可见、绑定和调用，不影响背后候选模型；撤销或封禁模型授权只影响模型广场可见、模型级 API Key 绑定、模型直调和业务接入点新增候选，不影响已授权公共接入点的内部路由。
- 两类授权互不自动产生，也不能互相扩大权限；接入点授权不隐含背后候选模型的模型级授权，封禁同理按资源独立生效。
- 租户业务接入点创建或编辑时只能选择本租户授权范围内的模型。
- 平台撤销或封禁某模型授权后，业务接入点中已有该模型候选进入不可用或异常状态；系统不自动替换成其他模型。
- 平台撤销授权或封禁后，已有 API Key 绑定保留，对应目标进入失效绑定状态；绑定不删除、API Key 不整体禁用；后续调用在授权层被拒。
- API Key 详情失效原因租户侧统一展示「平台撤销授权」，不区分撤权与封禁；平台侧审计区分撤权 / 封禁 / 解封三类动作。
- 平台重新授权或解封后，历史失效绑定自动恢复有效；不同资源不得自动复用旧绑定。
- 未授权或被封禁的资源在租户侧完全不出现：模型广场、接入点列表、API Key 绑定选择器的列表、筛选、汇总计数均不呈现。
- 存量公共资源上线时统一视同 `ALL` 且无黑名单条目，不生成租户维度记录，不做迁移，存量行为不变。
- 新发布公共资源在**资源授权范围**列表，默认ALL。
- `ALL` 与 `WHITELIST` 可事后双向切换，切换后立即影响后续调用链路判断。
- `ALL → WHITELIST` 切换时授权管理抽屉提供引导，支持按最近 30 天有调用的租户预填白名单，避免误伤存量调用方；切换与预填不影响黑名单条目。
- 调用目标为 `ep-*` 时只在授权层校验接入点的授权与黑名单，内部路由解析出的候选模型不再二次过授权层。
- 模型直调只校验模型的授权与黑名单。
- 公共接入点详情页套用 AI 安全护栏推荐配置时默认不启用；启用后进入观察模式。

## 8.1.3 边界条件与异常场景

### 输入边界

- 黑名单条目的 `reason` 字段为可选，允许为空提交。
- `visibility_mode` 值域仅限 `ALL` / `WHITELIST`，`NULL` 视同 `ALL`。

### 并发竞态

- 同一租户 × 资源的白名单与黑名单条目允许各一条并存（唯一约束），并发写入时由数据库唯一约束保证幂等。\[待确认:T-01] 并发撤权与封禁是否存在竞态导致绑定状态不一致，加用例验证。

### 降级容灾

- \[待确认:T-02] 缓存服务不可用时，授权校验是否降级为直接查库或拒绝调用。

### 存量兼容

- 存量公共资源升级后视同 `ALL` 且无黑名单条目，不生成任何租户维度迁移数据，行为不变。

### 缓存失效窗口

- 授权 / 封禁变更通过事件驱动失效 `tenant_id` 维度全量缓存，变更在下一次查询或调用校验中生效；失效窗口内存在短暂的不一致。

### 撤权保留失效绑定

- 撤权或封禁后 API Key 绑定保留、调用被拒，绑定表现为失效状态；重新授权或解封后自动恢复。

## 8.1.5 页面交互

### 操作入口

| 入口类型      | 菜单 / 位置         | 操作人   | 说明            |
| --------- | --------------- | ----- | ------------- |
| 主入口 / 事实源 | `运营管理 → 资源授权范围` | 平台管理员 | 授权与封禁关系的唯一事实源 |

租户管理员和普通用户无本页入口，也不提供租户侧「申请加入」路径；封禁事实（黑名单标记与 `reason`）仅平台侧可见。

### 交互流程

资源授权范围页面的资源来源与操作流程：

1. **资源自动同步**：创建公共接入点、纳管运营租户模型供应商模型、微调模型上架后，资源会自动同步到资源授权范围页面列表
2. **授权管理**：平台管理员在资源授权范围页面列表中选中目标资源，打开授权管理抽屉
3. **配置可见范围模式**：在授权管理抽屉中选择可见范围模式（`ALL` / `WHITELIST`），并管理白名单租户与黑名单租户
4. **保存生效**：保存授权配置后立即生效，租户侧可见性实时更新

### 资源授权范围页交互

- 双视角主视图：按资源（该资源授权/封禁了哪些租户）/ 按租户（该租户被授权/封禁了哪些资源）双向筛选查询(租户本期不做)
- 授权管理抽屉：维护单个资源的可见范围模式、白名单租户与黑名单租户（含 `reason`）；`ALL → WHITELIST` 切换引导见 8.1.2 产品规则
- 候选模型授权状态提示：接入点行展示背后候选模型的授权状态，仅作信息提示，不做校验拦截

### 租户侧展示结果

| 页面            | 展示规则                                        |
| ------------- | ------------------------------------------- |
| 模型广场          | 普通用户只展示本租户授权范围内（不在黑名单且 `ALL` 或在白名单）、状态可用的模型 |
| 接入点列表         | 普通用户只展示本租户授权范围内的公共接入点和自己可见的业务接入点            |
| API Key 绑定选择器 | 只展示本租户授权范围内的模型和公共接入点                        |

### 二次确认规则

- 撤权 / 封禁操作弹窗：展示影响清单（该租户关联有效 API Key 数、业务接入点候选引用、最近 30 天调用次数），确认后提交

### 提示 / 错误文案

| 场景                | 文案内容                                                          | 位置              |
| ----------------- | ------------------------------------------------------------- | --------------- |
| API Key 失效绑定      | 「平台撤销授权」                                                      | 租户侧 API Key 详情页 |
| 调用被授权层拒绝          | `RESOURCE_NOT_AUTHORIZED`（按 `Accept-Language` 返回三语言文案）        | 调用响应            |
| 调用被 API Key 权限层拒绝 | `RESOURCE_NOT_IN_KEY_PERMISSION`（按 `Accept-Language` 返回三语言文案） | 调用响应            |

## 8.1.6 状态流转

本功能无生命周期状态。租户授权范围是平台管理员对「租户 × 公共资源」的授权关系管理，授权关系本身可通过授权/撤权/封禁/解封动态增删，但资源本身不存在由本功能控制的状态机流转；资源的上下架状态由模型库和接入点各自的管理页面维护，不属于本功能职责。

## 8.1.7 验收标准

### 一、授权基础操作（AC-1.1 ~ AC-1.5）

- [ ] **AC-1.1** Given 公共接入点 ep-public 已创建且状态可用，租户 A 尚未被授权；When 平台管理员在授权管理抽屉中将 ep-public 授权给租户 A；Then 租户 A 普通用户立即在接入点列表可见 ep-public、可绑定 API Key、可通过 `ep-*` 调用。
- [ ] **AC-1.2** Given 模型 M 为公共模型且状态可用，租户 A 尚未被授权；When 平台管理员在授权管理抽屉中将模型 M 授权给租户 A；Then 租户 A 普通用户可在模型广场看到模型 M、可创建模型级 API Key、可作为业务接入点候选、可直调 `model=M`。
- [ ] **AC-1.3** Given 模型 M 为公共模型且未被授权给租户 B；When 租户 B 用户尝试查看模型广场、创建 API Key 绑定模型 M 或直调 `model=M`；Then 模型广场不可见 M、API Key 不可绑定 M、直调返回 `RESOURCE_NOT_AUTHORIZED`。
- [ ] **AC-1.4** Given 租户 A 已授权模型 M，存在公共接入点 ep-public 内部路由命中模型 M；When 平台管理员撤销租户 A 对模型 M 的授权；Then 租户 A 模型广场不展示 M、API Key 不能新增绑定 M、直调 `model=M` 被拒绝；ep-public 仍可按平台路由内部命中模型 M。
- [ ] **AC-1.5** Given 租户 A 已授权 ep-public，模型 M 也已授权给租户 A；When 平台管理员撤销租户 A 对 ep-public 的授权；Then 不联动撤销模型 M 的授权，模型 M 是否可见由模型授权单独决定。

### 二、资源自动同步与默认行为（AC-1.6 ~ AC-1.8）

- [ ] **AC-1.6** Given 公共接入点已创建；When 平台管理员未在资源授权范围页面执行授权操作；Then 接入点默认 `ALL` 模式，对所有租户可见。
- [ ] **AC-1.7** Given 公共资源 R 的可见范围模式为 `ALL`；When 资源 R 创建并自动同步到资源授权范围页面后；Then 默认对所有租户可见，无需逐租户配置授权记录。
- [ ] **AC-1.8** Given 新公共资源（模型或公共接入点）已创建并自动同步到资源授权范围页面；When 平台管理员在授权管理抽屉中保存配置；Then 必须显式选择可见范围模式（`ALL` 或 `WHITELIST`），未选择不能保存。

### 三、可见范围模式（AC-1.9 ~ AC-1.11）

- [ ] **AC-1.9** Given 模型 M 的可见范围模式为 `WHITELIST`，租户 B 未被列入白名单；When 租户 B 用户查看模型广场列表、筛选、汇总计数或 API Key 绑定选择器；Then 模型 M 完全不出现、不可绑定、直调返回 `RESOURCE_NOT_AUTHORIZED`。
- [ ] **AC-1.10** Given 公共资源 R 当前可见范围模式为 `ALL`；When 平台管理员将模式从 `ALL` 切换为 `WHITELIST`；Then 授权管理抽屉提供引导，支持按最近 30 天有调用的租户预填白名单；切换与预填不影响黑名单条目。
- [ ] **AC-1.11** Given 平台管理员在授权管理抽屉中配置可见范围模式与授权租户；When 保存配置后；Then 配置生效且之后独立演化，不影响背后候选模型的授权状态。

### 四、黑名单机制（AC-1.12 ~ AC-1.15）

- [ ] **AC-1.12** Given 公共资源 R 的可见范围模式为 `ALL`，租户 C 被加入黑名单；When 租户 C 用户尝试查看或调用资源 R；Then 全链路不可见、不可调用，返回 `RESOURCE_NOT_AUTHORIZED`；解封后自动恢复可见可调用。
- [ ] **AC-1.13** Given 公共资源 R 的可见范围模式为 `WHITELIST`，租户 D 在白名单中且被加入黑名单；When 租户 D 用户尝试查看或调用资源 R；Then 黑名单优先生效，租户 D 不可见不可调用；删除黑名单条目（解封）后，白名单资格与失效绑定自动恢复。
- [ ] **AC-1.14** Given 租户 E 被列入资源 R 的黑名单；When 租户 E 用户调用资源 R；Then 返回与未授权相同的错误码 `RESOURCE_NOT_AUTHORIZED`；封禁标记与 `reason` 仅平台侧授权页和审计可见。
- [ ] **AC-1.15** Given 公共接入点 ep-xxx 已授权给租户 A，其内部路由候选模型 M 未授权给租户 A；When 租户 A 用户通过 ep-xxx 发起调用；Then 只校验 ep-xxx 的授权与黑名单，候选模型 M 的授权状态不影响该调用。

### 五、API Key 失效与恢复（AC-1.16 ~ AC-1.17）

- [ ] **AC-1.16** Given 租户 A 已授权资源 R 且存在 API Key 绑定资源 R；When 平台管理员撤销授权或封禁资源 R；Then 已绑定 API Key 保留，租户侧统一展示「平台撤销授权」失效原因（不区分撤权与封禁）；重新授权或解封后，历史失效绑定自动恢复有效。
- [ ] **AC-1.17** Given 租户 A 当前已授权模型集合为 S；When 租户 A 用户创建或编辑业务接入点并选择候选模型；Then 只能选择集合 S 中的模型；已选模型被撤销授权后进入不可用/异常状态，系统不自动替换。

### 六、操作交互（AC-1.18 ~ AC-1.19）

- [ ] **AC-1.18** Given 平台管理员在授权管理抽屉中执行撤销授权或封禁租户操作；When 点击撤权/封禁按钮；Then 弹出二次确认弹窗，展示影响清单（关联有效 API Key 数、业务接入点候选引用、最近 30 天调用次数）。
- [ ] **AC-1.19** Given 资源授权范围页已打开；When 管理员切换视角；Then 支持按资源（该资源授权/封禁了哪些租户）和按租户（该租户被授权/封禁了哪些资源）双向筛选查询；接入点行展示候选模型授权状态信息提示。

### 七、缓存失效（AC-1.20）

- [ ] **AC-1.20** Given 授权/封禁变更已提交生效；When 变更事件触发；Then 对应租户缓存通过事件驱动失效，在下一次查询或调用校验中按最新数据生效。

### 八、网关校验与存量兼容（AC-1.21 ~ AC-1.22）

- [ ] **AC-1.21** Given 一个 API 调用请求到达网关；When 网关执行资源可见性校验；Then 按顺序执行：租户授权范围（先查黑名单，黑名单命中即拒；再按模式查白名单）→ API Key 权限范围 → 既有前置校验；首个失败立即返回对应错误码。
- [ ] **AC-1.22** Given 存量公共资源在系统升级前已存在；When 系统升级完成后；Then 行为不变（视同 `ALL`、无黑名单条目），未生成任何租户维度迁移数据。

## 8.1.8 补充约束

### 安全口径

- 黑名单拒绝与未授权对外返回相同错误码 `RESOURCE_NOT_AUTHORIZED`，调用方无法区分「从未授权」与「被封禁」。
- 封禁事实（黑名单标记与 `reason`）仅平台侧授权管理页可见，不对租户侧暴露。

### i18n

- 两条新错误码（`RESOURCE_NOT_AUTHORIZED` / `RESOURCE_NOT_IN_KEY_PERMISSION`）纳入 F16 网关错误码 i18n 文案映射表，按 `Accept-Language` 返回三语言文案，随 T17 首批清单交付。

### 数据同步

- Java 侧为权威事实源，沿用现有同步机制下发 Go 网关（含 `entry_type` 字段）。

### 权限门禁

- 资源授权范围页仅平台管理员角色可访问；授权管理抽屉从零实现，可复用 `model-choose-list` 抽屉范式与角色门禁。

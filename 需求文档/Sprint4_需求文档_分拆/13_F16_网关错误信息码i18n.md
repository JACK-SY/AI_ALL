# F16. 网关错误信息码 i18n

## 8.16.1 需求说明

`maix_ops_go`（new-api fork）在鉴权失败、路由失败、限额阻断、上游错误等场景返回错误码与错误信息。当前错误信息以中文（或部分英文）硬编码返回，客户端 / 开发者在不同语言偏好下看到同一份文案，`mop_docs` 三语言错误码文档也无法与网关运行时保持一致。本期为网关自身错误码引入 i18n，按客户端语言偏好返回对应文案。

## 8.16.2 产品规则

- 网关按错误码维护统一的错误文案映射表，支持简体中文（`zh-CN`）、繁体中文（`zh-TW`）、英文（`en`）三种语言。
- 语言偏好按 HTTP 请求头 `Accept-Language` 识别；缺失、不识别或未覆盖语言时缺省为简体中文。
- 网关自身错误码走 i18n 文案表；供应商上游返回的原始错误信息按透传处理，不强制翻译，避免与上游语义漂移。
- 未落到 i18n 映射的网关错误码，保留原始错误上下文透传，不返回空字符串或"未知错误"占位。
- 错误码 → 三语言文案映射表由 `mop_docs` 与网关共同维护，二者同源；文案调整通过 PR 同步更新，不各自维护副本。
- F1 增补的两条资源可见性错误码（`RESOURCE_NOT_AUTHORIZED` / `RESOURCE_NOT_IN_KEY_PERMISSION`，定义见 8.1.8.5）纳入首批 i18n 文案清单（T17）。
- i18n 仅覆盖错误信息的展示文案，`error_code` / `http_status` / 结构化 `error.type` 等契约字段保持一致，不因语言变化。
- 调用日志中记录网关返回的 `error_code` 与匹配到的 i18n 语言，供排障与文档核对。

## 8.16.3 边界条件与异常场景

- **Accept-Language 非标值解析规则**：
  - 标准格式 `en-US`、`zh-CN`、`zh-TW` 按主标签匹配（`en-US` → `en`，`zh-CN` → `zh-CN`，`zh-TW` → `zh-TW`）。
  - 带权重格式 `en-US,en;q=0.9,zh-CN;q=0.8` 按权重降序取第一个可匹配语言。
  - 仅主标签 `en`、`zh`：`en` → `en`；`zh` → `zh-CN`（缺省简体）。
  - 无法解析或空值 → `zh-CN`。
- **上游透传错误含非 UTF-8 字符**：网关对上游返回的原始错误信息按原字节透传，不做编码转换或截断；若原始信息含非 UTF-8 字符，保持原样透传，接收端负责解码。
- **映射表加载/热更新延迟**：i18n 映射表随网关进程启动加载至内存；PR 合并后需重启网关生效，本期不支持热更新。重启窗口期内新旧映射可能短暂共存。
- **调用日志 i18n 语言字段缺失**：若网关未成功匹配 `Accept-Language`，调用日志中 i18n 语言字段记录为 `default`，表示使用了缺省语言。

## 8.16.4 字段定义

### 页面展示字段 — 错误码映射表

| 字段 | 说明 | 是否本期新增 |
|------|------|:----------:|
| 错误码 | 平台定义的错误码标识（如 `RESOURCE_NOT_AUTHORIZED`） | 否（已有） |
| zh-CN | 简体中文错误文案 | 是 |
| zh-TW | 繁体中文错误文案 | 是 |
| en | 英文错误文案 | 是 |

### API 字段

| 字段 | 说明 | 是否本期新增 |
|------|------|:----------:|
| error.code | 错误码标识，不随语言变化 | 否（已有） |
| error.message | 按 i18n 匹配后的展示文案 | 是（新增 i18n 行为） |
| error.type | 结构化错误类型，不随语言变化 | 否（已有） |
| http_status | HTTP 状态码，不随语言变化 | 否（已有） |

### 数据库 / 索引字段

| 字段 | 说明 | 是否本期新增 | 存储 |
|------|------|:----------:|------|
| i18n_language | 调用日志中记录的匹配语言标识 | 是 | ES 查询索引 |
| error_code | 错误码标识 | 否（已有） | ES 查询索引 |

## 8.16.5 页面交互

### mop_docs 同步流程

- 错误码映射表以 JSON / YAML 文件形式存放于代码仓库，`mop_docs` 和网关项目共享同一份源文件。
- 新增错误码或修改文案时，开发者提交 PR 修改源文件；CI 流程校验三语言字段完备性（每个错误码须同时有 `zh-CN`/`zh-TW`/`en` 三个文案），缺少任一语言的 PR 会被 review 拦截。
- PR 合并后，`mop_docs` 文档站自动构建发布；网关需重启加载最新映射表。
- 平台管理界面本期不提供映射表的可视化编辑入口，仅通过代码仓库管理。

## 8.16.6 状态流转

本功能不涉及独立状态流转。i18n 行为为网关响应的即时逻辑，不改变错误码的生命周期。

## 8.16.7 验收标准

### 一、语言偏好解析（AC-16.1 ~ AC-16.3）

- [ ] **AC-16.1** Given 客户端请求携带 `Accept-Language=en`（或 `en-US`）；When 网关返回自身错误码；Then `error.message` 为英文文案，`error.code` / `http_status` / `error.type` 不变。
- [ ] **AC-16.2** Given 客户端请求携带 `Accept-Language=zh-TW`；When 网关返回自身错误码；Then `error.message` 为繁体中文文案。
- [ ] **AC-16.3** Given 客户端请求未携带 `Accept-Language` 或值为无法识别的语言；When 网关返回自身错误码；Then `error.message` 为简体中文文案。

### 二、错误透传（AC-16.4 ~ AC-16.5）

- [ ] **AC-16.4** Given 网关转发调用至上游供应商，上游返回错误响应；When 网关将上游错误透传返回客户端；Then 原始错误信息不被翻译，保留完整上下游错误上下文。
- [ ] **AC-16.5** Given 网关返回了一个尚未纳入 i18n 映射表的错误码；When 客户端收到响应；Then `error.message` 保留原始错误上下文，不返回空字符串或"未知错误"占位。

### 三、契约一致性（AC-16.6 ~ AC-16.7）

- [ ] **AC-16.6** Given `mop_docs` 文档站发布三语言错误码列表；When 对照网关运行时 i18n 映射表；Then 二者文案一致；PR 中未同步文档的错误码变更会被 CI 校验拦截。
- [ ] **AC-16.7** Given 同一错误码在不同 `Accept-Language` 下返回；When 对比不同语言下的响应；Then `error_code` / `http_status` / 结构化 `error.type` 完全一致，仅 `error.message` 文案不同。

### 四、权重解析与透传（AC-16.8 ~ AC-16.9）

- [ ] **AC-16.8** Given 客户端请求携带 `Accept-Language=en-US,en;q=0.9,zh-CN;q=0.8`；When 网关解析语言偏好；Then 按权重降序匹配，返回英文文案。
- [ ] **AC-16.9** Given 上游返回的原始错误信息中含有非 UTF-8 编码字符；When 网关透传返回；Then 原字节不被截断或转换，接收端负责解码。

### 五、日志记录（AC-16.10）

- [ ] **AC-16.10** Given 网关返回自身错误码且匹配了 `Accept-Language`；When 调用日志写入 ES 索引；Then 日志中记录 `error_code` 和 `i18n_language` 字段。

## 8.16.8 补充约束

- i18n 仅覆盖错误信息的展示文案，`error_code` / `http_status` / 结构化 `error.type` 等 API 契约字段保持一致，不因客户端语言偏好变化。
- 映射表源文件存放于代码仓库，不支持管理界面在线编辑。
- 首批 i18n 文案清单挂 T17 任务跟踪，包含 F1 增补的两条资源可见性错误码及 F8 AI 安全护栏拦截错误码。
- Accept-Language 解析逻辑对齐 RFC 7231 优先级规则。
